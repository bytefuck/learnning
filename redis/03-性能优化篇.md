# Redis面试题 - 性能优化篇

## 1. Redis内存占用过高时，应该如何排查和优化？

**解答：**

### 内存排查方法

#### 1. 查看内存使用情况

```bash
# 查看内存统计信息
redis-cli INFO memory

# 关键指标：
# used_memory: 1032120  # Redis分配器分配的内存总量
# used_memory_human: 1008.91K  # 人类可读格式
# used_memory_rss: 2052096  # 操作系统看到的物理内存占用
# used_memory_rss_human: 1.96M
# used_memory_peak: 1048576  # 内存使用峰值
# used_memory_lua: 37888  # Lua脚本占用的内存
# mem_fragmentation_ratio: 1.99  # 内存碎片率（used_memory_rss/used_memory）
```

#### 2. 分析内存使用详情

```bash
# 查看按数据类型统计的内存占用
redis-cli --bigkeys

# 查看具体key的内存占用（Redis 4.0+）
redis-cli MEMORY USAGE user:1000

# 抽样估算hash/set/zset的内存（SAMPLES指定抽样数量）
redis-cli MEMORY USAGE big_hash SAMPLES 5
```

#### 3. 使用rdb-tools分析RDB文件

```bash
# 安装工具
pip install rdbtools python-lzf

# 生成内存报告
rdb -c memory /var/lib/redis/dump.rdb > memory.csv

# 按内存使用量排序
sort -t, -k4 -nr memory.csv | head -20

# 输出格式：
# database,type,key,size_in_bytes,encoding,num_elements,len_largest_element
# 0,string,user:1000,1024,string,1,1000
```

### 内存优化策略

#### 1. 优化数据结构

**使用ziplist编码：**
```conf
# 调整ziplist阈值，使更多数据使用紧凑编码
hash-max-ziplist-entries 512
hash-max-ziplist-value 64
list-max-ziplist-size -2
zset-max-ziplist-entries 128
zset-max-ziplist-value 64
set-max-intset-entries 512
```

**使用整数存储：**
```bash
# 不好的做法 - 存储字符串数字
SET counter "12345"  # embstr/raw编码

# 好的做法 - 存储整数
SET counter 12345    # int编码，更省内存
```

**使用Hash存储小对象：**
```bash
# 不好的做法 - 每个字段一个key（10000个key）
SET user:1000:name "张三"
SET user:1000:age 25
SET user:1000:city "北京"

# 好的做法 - 使用Hash（1个key）
HSET user:1000 name "张三" age 25 city "北京"
# 内存节省约70%
```

**使用Bitmap/HyperLogLog：**
```bash
# 存储1亿用户的签到记录
# Set方式：约6GB
# Bitmap方式：约12MB
SETBIT sign:202312:user1000 0 1

# 存储1亿独立访客
# Set方式：约6GB
# HyperLogLog：12KB
PFADD uv:20231201 user1000
```

#### 2. 设置合理的过期时间

```bash
# 为临时数据设置过期时间
SET temp:data "value" EX 3600

# 使用expire命令
EXPIRE session:user1000 1800

# 定期清理脚本（避免使用keys命令）
redis-cli --scan --pattern "temp:*" | xargs redis-cli unlink
```

#### 3. 使用内存淘汰策略

```conf
# 配置最大内存
maxmemory 2gb

# 配置淘汰策略
maxmemory-policy allkeys-lru  # 所有key使用LRU淘汰
# 可选策略：
# - noeviction: 不淘汰，返回错误（默认）
# - allkeys-lru: 所有key使用LRU淘汰（推荐）
# - volatile-lru: 只淘汰设置了过期时间的key
# - allkeys-random: 随机淘汰所有key
# - volatile-random: 随机淘汰设置了过期时间的key
# - volatile-ttl: 淘汰即将过期的key
# - allkeys-lfu: 使用LFU算法（Redis 4.0+）
# - volatile-lfu: 对设置了过期时间的key使用LFU
```

#### 4. 数据压缩

```python
import gzip
import json
import redis

r = redis.Redis()

# 压缩存储
data = {'large': 'data' * 10000}
compressed = gzip.compress(json.dumps(data).encode())
r.set('key', compressed)

# 解压缩读取
compressed_data = r.get('key')
data = json.loads(gzip.decompress(compressed_data))
```

#### 5. 避免大key

```bash
# 发现大key
redis-cli --bigkeys

# 处理大key
# 1. 拆分大hash
HSCAN big_hash 0 COUNT 100  # 分批读取
HDEL big_hash field1 field2 ...  # 分批删除

# 2. 使用unlink异步删除大key
UNLINK big_key  # 不会阻塞

# 3. 分批处理列表
LRANGE big_list 0 99  # 每次读取100个
LTRIM big_list 100 -1  # 删除已读取的
```

## 2. Redis变慢了，应该从哪些方面排查？

**解答：**

### 排查步骤

#### 1. 检查慢查询

```bash
# 查看慢查询配置
redis-cli CONFIG GET slowlog-log-slower-than
redis-cli CONFIG GET slowlog-max-len

# 查看慢查询日志（默认超过10ms的命令）
redis-cli SLOWLOG GET 10

# 输出示例：
# 1) 1) (integer) 12345              # 唯一ID
#    2) (integer) 1640000000        # 执行时间戳
#    3) (integer) 15000             # 执行耗时（微秒）
#    4) 1) "keys"                   # 命令
#       2) "*"                     # 参数
```

**常见慢查询原因：**
```bash
# 1. 使用keys命令（禁止使用）
KEYS user:*  # O(n)，遍历所有key

# 替代方案
SCAN 0 MATCH user:* COUNT 100

# 2. 大key操作
HGETALL big_hash  # 返回百万级field
LRANGE big_list 0 -1  # 返回百万级元素

# 替代方案
HSCAN big_hash 0 COUNT 100
LRANGE big_list 0 99

# 3. 复杂聚合操作
SUNION store1 store2 store3  # 大量数据计算
SORT big_list BY weight_*  # 大数据排序

# 4. 删除大key
DEL big_hash  # 阻塞

# 替代方案
UNLINK big_hash  # 异步删除
```

#### 2. 检查系统资源

```bash
# 检查Redis进程资源占用
top -p $(pgrep redis-server)

# 检查系统负载
uptime

# 检查网络
redis-cli INFO stats | grep instantaneous_ops_per_sec
# 检查每秒操作数是否异常

# 检查连接数
redis-cli INFO clients
# connected_clients: 100
# client_longest_output_list: 0
# client_biggest_input_buf: 0
```

#### 3. 检查持久化影响

```bash
# 查看持久化状态
redis-cli INFO persistence

# 关键指标：
# rdb_last_bgsave_status: ok
# rdb_last_bgsave_time_sec: 5  # 上次bgsave耗时
# aof_last_rewrite_status: ok
# aof_last_rewrite_time_sec: 10  # 上次rewrite耗时

# 如果耗时过长，可能阻塞主线程
```

**优化方案：**
```conf
# 1. 使用AOF每秒刷盘（性能和安全的平衡）
appendfsync everysec

# 2. 开启AOF重写自动触发
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

# 3. 使用RDB+AOF混合持久化（Redis 4.0+）
aof-use-rdb-preamble yes

# 4. 在低峰期执行持久化
# 使用定时任务触发bgsave
```

#### 4. 检查内存交换

```bash
# 检查是否使用了swap
redis-cli INFO memory | grep swap

# 系统层面检查
redis-cli --latency-history -i 1
# 如果延迟波动很大，可能有swap

# Linux检查
cat /proc/$(pgrep redis-server)/status | grep Swap
```

**解决方案：**
```bash
# 1. 关闭swap
sudo swapoff -a

# 2. 调整内核参数
vm.swappiness = 1  # 尽量不使用swap

# 3. 增加物理内存或减少maxmemory
```

#### 5. 检查网络延迟

```bash
# 测试网络延迟
redis-cli --latency
# min: 0, max: 1, avg: 0.08 (单位毫秒)

# 历史延迟
redis-cli --latency-history -i 1
```

#### 6. 检查主从同步延迟

```bash
# 从节点查看
redis-cli INFO replication

# 关键指标：
# master_link_status: up
# master_last_io_seconds_ago: 2  # 距离上次与master通信秒数
# master_sync_in_progress: 0  # 是否正在全量同步

# 如果master_last_io_seconds_ago持续增大，说明主从延迟
```

### 常见性能瓶颈和解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| CPU 100% | 复杂命令、大量连接 | 优化命令、使用pipeline、增加分片 |
| 内存不足 | 数据量大、未设置过期 | 设置过期、开启淘汰、扩容 |
| 网络带宽满 | 大key读取、QPS过高 | 拆分大key、增加分片、本地缓存 |
| 磁盘IO高 | AOF每秒刷盘 | 改为no/always（权衡）、使用SSD |
| 阻塞 | 大key删除、持久化 | 使用unlink、优化持久化配置 |

## 3. 如何使用Pipeline和Lua脚本提升性能？

**解答：**

### Pipeline的使用

#### 原理

```
普通模式（100次操作）：
客户端 → 命令1 → 服务端 → 响应1 → 客户端
客户端 → 命令2 → 服务端 → 响应2 → 客户端
...
客户端 → 命令100 → 服务端 → 响应100 → 客户端
总RTT = 100 * 网络往返时间

Pipeline模式（100次操作）：
客户端 → 命令1 → 
      → 命令2 → 
      → ... → 
      → 命令100 → 服务端
服务端 → 响应1 → 
      → 响应2 → 
      → ... → 
      → 响应100 → 客户端
总RTT = 1 * 网络往返时间 + 处理时间
```

#### 使用示例

**Java代码：**
```java
@Service
public class PipelineService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    // 批量设置（不使用pipeline）
    public void batchSetWithoutPipeline(Map<String, String> data) {
        for (Map.Entry<String, String> entry : data.entrySet()) {
            redisTemplate.opsForValue().set(entry.getKey(), entry.getValue());
        }
        // 1000次操作 = 1000次网络往返
    }
    
    // 批量设置（使用pipeline）
    public void batchSetWithPipeline(Map<String, String> data) {
        redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
            for (Map.Entry<String, String> entry : data.entrySet()) {
                connection.set(entry.getKey().getBytes(), 
                    entry.getValue().getBytes());
            }
            return null;
        });
        // 1000次操作 = 1次网络往返
    }
    
    // 批量读取
    public List<String> batchGetWithPipeline(List<String> keys) {
        List<Object> results = redisTemplate.executePipelined(
            (RedisCallback<Object>) connection -> {
                for (String key : keys) {
                    connection.get(key.getBytes());
                }
                return null;
            }
        );
        
        return results.stream()
            .map(obj -> obj != null ? obj.toString() : null)
            .collect(Collectors.toList());
    }
    
    // 混合操作
    public void mixedOperationsPipeline() {
        redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
            // 读取
            connection.get("key1".getBytes());
            connection.hGetAll("hash1".getBytes());
            
            // 写入
            connection.set("key2".getBytes(), "value2".getBytes());
            connection.hSet("hash2".getBytes(), "field".getBytes(), 
                "value".getBytes());
            
            // 计数
            connection.incr("counter".getBytes());
            
            return null;
        });
    }
}
```

**性能对比：**
```
测试环境：本地Redis，10000次操作

不使用Pipeline：
- 耗时：约5000ms
- 每秒操作：2000次

使用Pipeline：
- 耗时：约100ms
- 每秒操作：100000次
- 性能提升：50倍
```

#### 注意事项

```java
// 1. pipeline不是原子操作（命令可能部分执行）
// 2. 单个pipeline命令数不宜过多（建议100-10000）
// 3. 数据量过大可能阻塞Redis

// 分批处理大数据量
public void batchProcessLargeData(List<String> data) {
    int batchSize = 1000;
    for (int i = 0; i < data.size(); i += batchSize) {
        List<String> batch = data.subList(i, 
            Math.min(i + batchSize, data.size()));
        
        redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
            for (String item : batch) {
                connection.rPush("queue".getBytes(), item.getBytes());
            }
            return null;
        });
    }
}
```

### Lua脚本的使用

#### 原理

```
Lua脚本在Redis中执行的特点：
1. 原子性：整个脚本作为一个命令执行，不会被其他命令打断
2. 减少网络往返：多个操作在一个脚本中完成
3. 复用：脚本可以被缓存，重复执行无需重新传输

执行流程：
客户端 → EVAL脚本 → Redis
Redis → 执行脚本（原子性）→ 返回结果 → 客户端
```

#### 使用示例

**原子性操作：**
```java
@Service
public class LuaScriptService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    // 原子性扣减库存
    private static final String DEDUCT_STOCK_SCRIPT = 
        "local stock = tonumber(redis.call('get', KEYS[1])); " +
        "if stock == nil then return -1 end; " +
        "if stock < tonumber(ARGV[1]) then return 0 end; " +
        "redis.call('decrby', KEYS[1], ARGV[1]); " +
        "return redis.call('get', KEYS[1]);";
    
    public Long deductStock(String productId, int quantity) {
        String key = "stock:" + productId;
        
        DefaultRedisScript<Long> script = new DefaultRedisScript<>();
        script.setScriptText(DEDUCT_STOCK_SCRIPT);
        script.setResultType(Long.class);
        
        return redisTemplate.execute(script, 
            Collections.singletonList(key), 
            String.valueOf(quantity));
        // 返回：-1=无库存，0=库存不足，>0=剩余库存
    }
    
    // 原子性释放分布式锁
    private static final String UNLOCK_SCRIPT = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "return redis.call('del', KEYS[1]) " +
        "else return 0 end";
    
    public boolean unlock(String lockKey, String requestId) {
        DefaultRedisScript<Long> script = new DefaultRedisScript<>();
        script.setScriptText(UNLOCK_SCRIPT);
        script.setResultType(Long.class);
        
        Long result = redisTemplate.execute(script,
            Collections.singletonList(lockKey),
            requestId);
        
        return result != null && result > 0;
    }
    
    // 批量操作（原子性）
    private static final String BATCH_UPDATE_SCRIPT = 
        "for i=1,#KEYS do " +
        "  redis.call('set', KEYS[i], ARGV[i]); " +
        "end " +
        "return #KEYS";
    
    public int batchUpdate(Map<String, String> data) {
        List<String> keys = new ArrayList<>(data.keySet());
        List<String> values = new ArrayList<>(data.values());
        
        DefaultRedisScript<Long> script = new DefaultRedisScript<>();
        script.setScriptText(BATCH_UPDATE_SCRIPT);
        script.setResultType(Long.class);
        
        Long result = redisTemplate.execute(script, keys, values.toArray());
        return result != null ? result.intValue() : 0;
    }
}
```

**复杂业务逻辑：**
```lua
-- 限制用户在一定时间内的操作次数
-- KEYS[1]: 计数器key
-- ARGV[1]: 时间窗口（秒）
-- ARGV[2]: 最大次数
-- ARGV[3]: 当前时间戳

local key = KEYS[1]
local window = tonumber(ARGV[1])
local max_count = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

-- 清理过期数据
redis.call('zremrangebyscore', key, 0, now - window)

-- 获取当前次数
local count = redis.call('zcard', key)

if count < max_count then
    -- 添加本次操作
    redis.call('zadd', key, now, now)
    -- 设置过期时间
    redis.call('expire', key, window)
    return {1, count + 1}  -- 允许，返回当前次数
else
    return {0, count}  -- 拒绝，返回当前次数
end
```

```java
public RateLimitResult rateLimit(String userId, int maxRequests, int windowSeconds) {
    String key = "rate_limit:" + userId;
    long now = System.currentTimeMillis();
    
    String script = 
        "local key = KEYS[1] " +
        "local window = tonumber(ARGV[1]) " +
        "local max_count = tonumber(ARGV[2]) " +
        "local now = tonumber(ARGV[3]) " +
        "redis.call('zremrangebyscore', key, 0, now - window * 1000) " +
        "local count = redis.call('zcard', key) " +
        "if count < max_count then " +
        "  redis.call('zadd', key, now, now) " +
        "  redis.call('expire', key, window) " +
        "  return {1, count + 1} " +
        "else " +
        "  return {0, count} " +
        "end";
    
    DefaultRedisScript<List> redisScript = new DefaultRedisScript<>();
    redisScript.setScriptText(script);
    redisScript.setResultType(List.class);
    
    List<Long> result = redisTemplate.execute(redisScript,
        Collections.singletonList(key),
        String.valueOf(windowSeconds),
        String.valueOf(maxRequests),
        String.valueOf(now));
    
    boolean allowed = result.get(0) == 1;
    int currentCount = result.get(1).intValue();
    
    return new RateLimitResult(allowed, currentCount, maxRequests);
}
```

## 4. 如何设计Redis缓存架构？缓存穿透、击穿、雪崩怎么解决？

**解答：**

### 缓存架构设计

#### 经典三层架构

```
┌─────────────────────────────────────┐
│           Application               │
└──────────────┬──────────────────────┘
               │
               ↓
┌─────────────────────────────────────┐
│        Local Cache (Caffeine)       │
│  - 热点数据本地缓存                   │
│  - TTL: 1-5分钟                      │
│  - 减少Redis压力                     │
└──────────────┬──────────────────────┘
               │ miss
               ↓
┌─────────────────────────────────────┐
│         Redis Cluster               │
│  - 热点数据分布式缓存                 │
│  - TTL: 5-30分钟                     │
│  - 主从复制保证高可用                  │
└──────────────┬──────────────────────┘
               │ miss
               ↓
┌─────────────────────────────────────┐
│         Database (MySQL)            │
│  - 持久化存储                        │
│  - 主从分离                          │
└─────────────────────────────────────┘
```

#### 数据一致性问题

**Cache-Aside模式（旁路缓存）：**
```
读操作：
1. 读缓存
2. 缓存命中，直接返回
3. 缓存未命中，读数据库
4. 写入缓存，设置过期时间
5. 返回数据

写操作：
1. 更新数据库
2. 删除缓存（不是更新缓存）
3. 为什么删除而不是更新？
   - 避免并发写导致数据不一致
   - 延迟双删策略：先删缓存→更新数据库→再删缓存
```

**Java实现：**
```java
@Service
public class CacheService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private UserDao userDao;
    
    @Autowired
    private Cache<String, User> localCache;
    
    private static final long REDIS_TTL = 30;
    private static final long LOCAL_TTL = 5;
    
    // 读取数据
    public User getUser(Long userId) {
        String key = "user:" + userId;
        
        // 1. 先查本地缓存
        User user = localCache.getIfPresent(key);
        if (user != null) {
            return user;
        }
        
        // 2. 再查Redis
        String json = redisTemplate.opsForValue().get(key);
        if (json != null) {
            user = JSON.parseObject(json, User.class);
            // 回填本地缓存
            localCache.put(key, user);
            return user;
        }
        
        // 3. 查数据库
        user = userDao.selectById(userId);
        if (user != null) {
            // 写入Redis
            redisTemplate.opsForValue().set(key, 
                JSON.toJSONString(user), 
                REDIS_TTL, TimeUnit.MINUTES);
            // 回填本地缓存
            localCache.put(key, user);
        }
        
        return user;
    }
    
    // 更新数据（延迟双删）
    @Transactional
    public void updateUser(User user) {
        String key = "user:" + user.getId();
        
        // 1. 删除缓存
        redisTemplate.delete(key);
        localCache.invalidate(key);
        
        // 2. 更新数据库
        userDao.updateById(user);
        
        // 3. 延迟再次删除缓存（解决并发问题）
        CompletableFuture.runAsync(() -> {
            try {
                Thread.sleep(500);  // 延迟500ms
                redisTemplate.delete(key);
                localCache.invalidate(key);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });
    }
}
```

### 缓存穿透

**问题描述：**
```
大量请求查询一个不存在的数据：
1. 缓存中没有（一定不会命中）
2. 数据库中也没有
3. 每次请求都打到数据库
4. 可能导致数据库崩溃

常见原因：
- 恶意攻击（查询不存在的id）
- 业务代码bug（查询无效参数）
- 数据被删除但未更新缓存
```

**解决方案：**

**1. 布隆过滤器（Bloom Filter）**
```java
@Component
public class BloomFilterService {
    
    @Autowired
    private RedissonClient redissonClient;
    
    private RBloomFilter<Long> bloomFilter;
    
    @PostConstruct
    public void init() {
        bloomFilter = redissonClient.getBloomFilter("userFilter");
        // 预期插入100万数据，误判率0.01
        bloomFilter.tryInit(1000000L, 0.01);
        
        // 预热：加载所有用户ID
        loadAllUserIds();
    }
    
    public void add(Long userId) {
        bloomFilter.add(userId);
    }
    
    public boolean mightContain(Long userId) {
        return bloomFilter.contains(userId);
    }
    
    // 查询时先判断
    public User getUserWithBloomFilter(Long userId) {
        // 布隆过滤器判断
        if (!mightContain(userId)) {
            return null;  // 一定不存在
        }
        
        // 可能存在，继续查询缓存和数据库
        return getUser(userId);
    }
}
```

### 缓存击穿

**问题描述：**
```
热点key突然失效，大量请求同时打到数据库：
1. 某个热点key过期
2. 大量并发请求同时查询这个key
3. 所有请求都打到数据库
4. 可能导致数据库崩溃
```

**解决方案：**

**1. 互斥锁（分布式锁）**
```java
public User getUserWithMutex(Long userId) {
    String key = "user:" + userId;
    String lockKey = "lock:user:" + userId;
    
    // 1. 查缓存
    String json = redisTemplate.opsForValue().get(key);
    if (json != null) {
        return JSON.parseObject(json, User.class);
    }
    
    // 2. 获取分布式锁
    String requestId = UUID.randomUUID().toString();
    boolean locked = tryLock(lockKey, requestId, 10);
    
    if (!locked) {
        // 获取锁失败，等待后重试
        sleep(100);
        return getUserWithMutex(userId);  // 递归重试
    }
    
    try {
        // 双重检查（其他线程可能已加载）
        json = redisTemplate.opsForValue().get(key);
        if (json != null) {
            return JSON.parseObject(json, User.class);
        }
        
        // 3. 查数据库
        User user = userDao.selectById(userId);
        if (user != null) {
            redisTemplate.opsForValue().set(key, 
                JSON.toJSONString(user), 30, TimeUnit.MINUTES);
        }
        return user;
    } finally {
        unlock(lockKey, requestId);
    }
}
```

**2. 逻辑过期（永不过期）**
```java
@Data
public class RedisData {
    private LocalDateTime expireTime;  // 逻辑过期时间
    private Object data;               // 实际数据
}

public User getUserWithLogicalExpire(Long userId) {
    String key = "user:" + userId;
    
    String json = redisTemplate.opsForValue().get(key);
    if (json == null) {
        return null;
    }
    
    RedisData redisData = JSON.parseObject(json, RedisData.class);
    User user = (User) redisData.getData();
    
    // 判断是否逻辑过期
    if (redisData.getExpireTime().isAfter(LocalDateTime.now())) {
        return user;  // 未过期，直接返回
    }
    
    // 已过期，需要重建缓存
    // 开启独立线程重建缓存
    CompletableFuture.runAsync(() -> {
        String lockKey = "lock:user:" + userId;
        String requestId = UUID.randomUUID().toString();
        
        if (tryLock(lockKey, requestId, 10)) {
            try {
                // 双重检查
                String newJson = redisTemplate.opsForValue().get(key);
                RedisData newData = JSON.parseObject(newJson, RedisData.class);
                if (newData.getExpireTime().isAfter(LocalDateTime.now())) {
                    return;  // 其他线程已更新
                }
                
                // 重建缓存
                User dbUser = userDao.selectById(userId);
                RedisData newRedisData = new RedisData();
                newRedisData.setData(dbUser);
                newRedisData.setExpireTime(LocalDateTime.now().plusMinutes(30));
                
                redisTemplate.opsForValue().set(key, 
                    JSON.toJSONString(newRedisData));
            } finally {
                unlock(lockKey, requestId);
            }
        }
    });
    
    // 返回旧数据（过期但可用）
    return user;
}
```

### 缓存雪崩

**问题描述：**
```
大量key同时过期，或Redis宕机：
1. 大量key设置相同的过期时间
2. 某一时刻大量key同时失效
3. 所有请求打到数据库
4. 数据库压力过大崩溃
```

**解决方案：**

**1. 过期时间加随机值**
```java
public void setWithRandomExpire(String key, Object value, long baseExpireMinutes) {
    // 基础过期时间 + 随机0-5分钟
    long expireMinutes = baseExpireMinutes + ThreadLocalRandom.current().nextInt(5);
    redisTemplate.opsForValue().set(key, JSON.toJSONString(value), 
        expireMinutes, TimeUnit.MINUTES);
}

// 批量设置时
public void batchSet(Map<String, User> users) {
    for (Map.Entry<String, User> entry : users.entrySet()) {
        // 过期时间分散在30-35分钟之间
        setWithRandomExpire(entry.getKey(), entry.getValue(), 30);
    }
}
```

**2. 多级缓存**
```java
@Service
public class MultiLevelCacheService {
    
    @Autowired
    private Cache<String, Object> localCache;  // Caffeine本地缓存
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    // 本地缓存永不过期（或设置很长），Redis缓存正常过期
    public Object get(String key) {
        // 1. 本地缓存（永不过期）
        Object value = localCache.getIfPresent(key);
        if (value != null) {
            return value;
        }
        
        // 2. Redis缓存
        String json = redisTemplate.opsForValue().get(key);
        if (json != null) {
            value = JSON.parse(json);
            localCache.put(key, value);
            return value;
        }
        
        // 3. 数据库
        value = loadFromDatabase(key);
        if (value != null) {
            redisTemplate.opsForValue().set(key, JSON.toJSONString(value), 
                30, TimeUnit.MINUTES);
            localCache.put(key, value);
        }
        
        return value;
    }
}
```

```java
@Component
public class CacheCircuitBreaker {
    
    private CircuitBreaker circuitBreaker;
    
    @PostConstruct
    public void init() {
        circuitBreaker = CircuitBreaker.ofDefaults("redis");
    }
    
    public Object executeWithFallback(String key, Supplier<Object> cacheSupplier, 
            Supplier<Object> fallbackSupplier) {
        return circuitBreaker.executeSupplier(() -> {
            try {
                return cacheSupplier.get();
            } catch (Exception e) {
                // Redis故障，返回降级数据
                return fallbackSupplier.get();
            }
        });
    }
}

// 使用示例
public User getUserWithCircuitBreaker(Long userId) {
    return (User) circuitBreaker.executeWithFallback(
        "user:" + userId,
        () -> getUserFromRedis(userId),
        () -> {
            // 降级：直接查数据库或返回默认值
            log.warn("Redis fallback, query DB directly");
            return userDao.selectById(userId);
        }
    );
}
```

## 5. 如何保证Redis与数据库的数据一致性？

**解答：**

### 一致性问题的产生

```
并发场景下的数据不一致：

线程A读取数据 → 缓存未命中 → 读取数据库（旧值）
线程B更新数据 → 更新数据库（新值） → 删除缓存
线程A写入缓存（旧值）

结果：缓存中是旧值，数据库是新值
```

### 解决方案

#### 1. Cache-Aside模式 + 延迟双删

**标准流程：**
```java
@Service
public class DataConsistencyService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private UserDao userDao;
    
    // 读取
    public User getUser(Long userId) {
        String key = "user:" + userId;
        
        // 先查缓存
        String json = redisTemplate.opsForValue().get(key);
        if (json != null) {
            return JSON.parseObject(json, User.class);
        }
        
        // 查数据库
        User user = userDao.selectById(userId);
        if (user != null) {
            // 写入缓存
            redisTemplate.opsForValue().set(key, 
                JSON.toJSONString(user), 30, TimeUnit.MINUTES);
        }
        
        return user;
    }
    
    // 更新 - 延迟双删
    @Transactional
    public void updateUser(User user) {
        String key = "user:" + user.getId();
        
        // 1. 先删缓存
        redisTemplate.delete(key);
        
        // 2. 更新数据库
        userDao.updateById(user);
        
        // 3. 延迟删缓存（异步）
        CompletableFuture.runAsync(() -> {
            try {
                Thread.sleep(500);  // 延迟500ms
                redisTemplate.delete(key);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });
    }
}
```

**为什么先删缓存而不是先更新数据库？**
```
方案1：先更新数据库，再删缓存
问题：
线程A更新数据库 → 线程B读取旧缓存 → 线程A删除缓存
结果：线程B读到了旧数据，但数据很快不一致

方案2：先删缓存，再更新数据库
问题：
线程A删除缓存 → 线程B读取数据库（旧值） → 线程A更新数据库 → 线程B写入旧值到缓存
结果：缓存中是旧值，数据库是新值

解决方案：延迟双删
线程A删除缓存 → 线程A更新数据库 → 延迟 → 线程A再次删除缓存
效果：即使线程B在间隙写入了旧值，也会被第二次删除清除
```

#### 2. 读写锁（Read-Through/Write-Through）

```java
@Component
public class ReadWriteLockCache {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private RedissonClient redissonClient;
    
    // 使用Redisson读写锁
    public User getUserWithLock(Long userId) {
        String key = "user:" + userId;
        RReadWriteLock rwLock = redissonClient.getReadWriteLock("rwLock:" + userId);
        
        // 读锁
        RLock readLock = rwLock.readLock();
        try {
            readLock.lock();
            
            // 查缓存
            String json = redisTemplate.opsForValue().get(key);
            if (json != null) {
                return JSON.parseObject(json, User.class);
            }
            
            // 升级写锁
            readLock.unlock();
            RLock writeLock = rwLock.writeLock();
            writeLock.lock();
            
            try {
                // 双重检查
                json = redisTemplate.opsForValue().get(key);
                if (json != null) {
                    return JSON.parseObject(json, User.class);
                }
                
                // 查数据库并写入缓存
                User user = userDao.selectById(userId);
                if (user != null) {
                    redisTemplate.opsForValue().set(key, 
                        JSON.toJSONString(user), 30, TimeUnit.MINUTES);
                }
                return user;
            } finally {
                writeLock.unlock();
            }
        } finally {
            if (readLock.isHeldByCurrentThread()) {
                readLock.unlock();
            }
        }
    }
}
```

#### 3. 基于Canal的异步更新

**架构：**
```
MySQL → Canal Server → Kafka → 消费者 → Redis

流程：
1. 应用更新数据库
2. Canal监控binlog，发现数据变更
3. Canal发送变更消息到Kafka
4. 消费者消费消息，更新/删除Redis缓存
```

**Java实现：**
```java
@Component
public class CanalConsumer {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @KafkaListener(topics = "canal_topic")
    public void onMessage(ConsumerRecord<String, String> record) {
        CanalMessage message = JSON.parseObject(record.value(), CanalMessage.class);
        
        if ("user".equals(message.getTable())) {
            for (CanalRowData rowData : message.getData()) {
                Long userId = rowData.getUserId();
                
                // 根据操作类型处理
                if ("UPDATE".equals(message.getType())) {
                    // 删除缓存，下次读取时重建
                    String key = "user:" + userId;
                    redisTemplate.delete(key);
                } else if ("DELETE".equals(message.getType())) {
                    // 删除缓存
                    String key = "user:" + userId;
                    redisTemplate.delete(key);
                } else if ("INSERT".equals(message.getType())) {
                    // 新数据，可以预加载到缓存
                    User user = convertToUser(rowData);
                    String key = "user:" + userId;
                    redisTemplate.opsForValue().set(key, 
                        JSON.toJSONString(user), 30, TimeUnit.MINUTES);
                }
            }
        }
    }
}
```

**优点：**
- 最终一致性
- 业务代码无侵入
- 解耦缓存更新逻辑

**缺点：**
- 有一定延迟（秒级）
- 系统复杂度高

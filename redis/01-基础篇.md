# Redis面试题 - 基础篇

## 1. 什么是Redis？它有哪些特点？

**解答：**

Redis（Remote Dictionary Server）是一个开源的高性能键值对（key-value）存储数据库，它支持多种数据结构，包括字符串（String）、哈希（Hash）、列表（List）、集合（Set）、有序集合（Sorted Set）等。Redis的数据存储在内存中，因此具有极高的读写性能。

**核心特点：**

1. **内存存储**：所有数据都存储在内存中，读写速度极快，能够达到每秒10万+的QPS（每秒查询率）。

2. **丰富的数据结构**：不仅仅支持简单的key-value，还支持复杂的数据结构，如列表、集合、有序集合、位图、HyperLogLog、地理空间等。

3. **持久化**：支持RDB（快照）和AOF（追加文件）两种持久化方式，可以将内存中的数据保存到磁盘，防止数据丢失。

4. **高可用性**：通过主从复制（Replication）、哨兵（Sentinel）和集群（Cluster）机制实现高可用。

5. **原子性操作**：所有操作都是原子性的，这意味着操作要么全部执行成功，要么全部不执行，非常适合用于并发控制。

6. **发布/订阅功能**：支持pub/sub模式，可以实现消息队列功能。

7. **Lua脚本支持**：支持使用Lua脚本编写复杂的原子性操作。

8. **事务支持**：提供MULTI/EXEC命令实现简单的事务功能。

## 2. Redis和Memcached的区别是什么？

**解答：**

Redis和Memcached都是高性能的内存数据库，但它们在多个方面存在显著差异：

### 数据类型支持

**Memcached**：
- 仅支持简单的key-value存储
- 值的类型只能是字符串
- 不支持复杂的数据结构操作

**Redis**：
- 支持String、Hash、List、Set、Sorted Set等多种数据类型
- 每种数据类型都有丰富的操作命令
- 支持Bitmaps、HyperLogLog、GEO等高级数据结构

### 数据持久化

**Memcached**：
- 不支持数据持久化
- 服务器重启后数据全部丢失
- 仅作为缓存使用

**Redis**：
- 支持RDB快照持久化
- 支持AOF日志持久化
- 可以配置自动保存策略
- 支持RDB+AOF混合持久化（Redis 4.0+）

### 内存管理机制

**Memcached**：
- 使用Slab Allocation机制管理内存
- 将内存分割成不同大小的块，减少碎片
- 采用LRU（Least Recently Used）算法淘汰数据
- 不支持内存超出后的处理策略配置

**Redis**：
- 使用jemalloc内存分配器
- 支持多种内存淘汰策略：
  - noeviction：不淘汰，直接返回错误
  - allkeys-lru：在所有key中使用LRU淘汰
  - volatile-lru：在设置了过期时间的key中使用LRU淘汰
  - allkeys-random：在所有key中随机淘汰
  - volatile-random：在设置了过期时间的key中随机淘汰
  - volatile-ttl：淘汰即将过期的key
  - allkeys-lfu/volatile-lfu（Redis 4.0+）：使用LFU算法

### 线程模型

**Memcached**：
- 多线程架构
- 主线程监听连接，工作线程处理请求
- 可以充分利用多核CPU

**Redis**：
- 单线程模型（Redis 6.0之前）
- 使用I/O多路复用技术（epoll/kqueue/select）处理并发
- 避免了多线程的上下文切换和锁竞争
- Redis 6.0引入多线程I/O，但命令执行仍是单线程

### 集群和高可用

**Memcached**：
- 不支持内置集群
- 需要客户端实现一致性哈希来分片
- 无高可用机制，服务器故障会导致数据丢失

**Redis**：
- 支持主从复制（Replication）
- 支持哨兵（Sentinel）实现自动故障转移
- 支持Redis Cluster实现数据分片和高可用
- 提供完整的高可用解决方案

### 应用场景

**Memcached**：
- 纯缓存场景
- 简单key-value存储
- 数据可以丢失的场景
- 多核CPU环境

**Redis**：
- 缓存场景
- 消息队列（List/Stream）
- 计数器、排行榜（Sorted Set）
- 分布式锁
- 会话存储
- 实时数据分析
- 地理位置应用

## 3. 为什么Redis这么快？

**解答：**

Redis的高性能（每秒可执行超过10万次读写操作）源于以下多个因素：

### 1. 完全基于内存操作

```
传统数据库：磁盘I/O → 内存 → CPU处理
Redis：内存 → CPU处理（跳过了磁盘I/O）
```

内存访问速度与磁盘访问速度的对比：
- 内存访问延迟：约100纳秒
- SSD磁盘访问延迟：约100微秒（是内存的1000倍）
- 机械硬盘访问延迟：约10毫秒（是内存的100000倍）

### 2. 单线程架构

**Redis 6.0之前的单线程模型：**

```
┌─────────────────────────────────────┐
│           单线程事件循环              │
│  ┌─────────┐    ┌───────────────┐   │
│  │  监听   │───→│  事件处理器   │   │
│  │  socket │    │  (epoll/kqueue)│  │
│  └─────────┘    └───────┬───────┘   │
│                         ↓           │
│                  ┌─────────────┐     │
│                  │ 命令执行    │     │
│                  │ (单线程)    │     │
│                  └─────────────┘     │
└─────────────────────────────────────┘
```

**单线程的优势：**

1. **无锁竞争**：不需要为数据操作加锁，避免了锁竞争带来的性能损耗
2. **无上下文切换**：线程切换需要保存和恢复寄存器、程序计数器等状态，单线程避免了这部分开销
3. **原子性保证**：所有命令都是原子执行，不会出现并发问题
4. **代码简单**：单线程模型使代码更简单，减少了多线程编程的复杂性

**为什么单线程不会成为瓶颈？**

Redis的瓶颈不在于CPU，而在于内存和网络带宽。大多数操作都是O(1)复杂度，执行速度极快，CPU不是限制因素。

### 3. I/O多路复用

Redis使用epoll（Linux）、kqueue（BSD/macOS）、select（Windows）等I/O多路复用技术：

```
传统多线程模型：
- 每个连接一个线程
- 10000个连接 = 10000个线程
- 大量上下文切换和内存开销

Redis单线程+多路复用模型：
- 一个线程管理所有连接
- 使用epoll监控所有socket的可读可写事件
- 只有真正有数据时才进行读写
- 10000个连接 = 1个线程
```

epoll的优势：
- 没有最大连接数限制（仅受限于系统内存）
- 使用事件通知，不需要轮询
- 时间复杂度O(1)，与连接数无关

### 4. 高效的数据结构

Redis针对不同的使用场景设计了专门的数据结构：

```
SDS（Simple Dynamic String）- 简单动态字符串：
- 预分配空间，减少内存重分配次数
- 记录长度信息，O(1)时间获取字符串长度
- 二进制安全，可以存储任意数据

Skip List（跳表）- 用于Sorted Set：
- 实现简单，性能接近红黑树
- 支持范围查询，时间复杂度O(log n)
- 插入、删除、查找都是O(log n)

ZipList（压缩列表）- 小规模数据：
- 连续内存存储，缓存友好
- 节省内存空间
- 适合存储小对象和少量数据

IntSet（整数集合）- 整数集合：
- 当集合中只有整数时使用
- 根据数据范围自动选择编码（int16/int32/int64）
- 内存效率高
```

### 5. 底层优化

**VM机制**（Redis 2.6之前）：
- 将不常访问的数据换出到磁盘
- 2.6之后废弃，推荐使用Redis Cluster进行水平扩展

**内存优化**：
- 使用jemalloc内存分配器，减少内存碎片
- 对象共享：小整数（0-9999）和常用字符串共享内存
- 编码转换：根据数据大小自动选择最优编码

### 6. Redis 6.0的多线程优化

Redis 6.0引入了多线程I/O，但命令执行仍然是单线程：

```
┌─────────────────────────────────────┐
│           I/O线程池                 │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐   │
│  │线程1│ │线程2│ │线程3│ │线程4│   │
│  └──┬──┘ └──┬──┘ └──┬──┘ └──┬──┘   │
│     └───────┴───────┴───────┘      │
│                 ↓                   │
│         网络数据的读写               │
│                 ↓                   │
│  ┌─────────────────────────────┐    │
│  │      主线程（单线程）        │    │
│  │    解析命令、执行命令         │    │
│  │    生成响应、发送响应         │    │
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘
```

多线程I/O的优势：
- 将网络数据的读写从主线程剥离
- 充分利用多核CPU处理网络I/O
- 命令执行仍保持单线程，保证原子性
- 在大型实例和网络密集型场景下性能提升明显

## 4. Redis的线程模型是怎样的？

**解答：**

### Redis 6.0之前的单线程模型

Redis 6.0之前采用经典的单线程事件驱动模型：

**执行流程：**

```
1. 初始化阶段
   ↓
2. 创建事件循环（aeEventLoop）
   ↓
3. 注册监听socket到事件循环
   ↓
4. 进入事件循环（aeMain）
   ↓
5. 等待事件发生（aeProcessEvents）
   ├── 文件事件（File Events）：可读、可写
   │   ├── 客户端连接请求 → 接受连接（accept）
   │   ├── 客户端命令请求 → 读取数据（read）→ 执行命令 → 发送响应（write）
   │   └── 主从复制数据 → 处理复制流
   │
   └── 时间事件（Time Events）：定时任务
       ├── 过期键清理
       ├── RDB持久化定时保存
       ├── AOF重写
       └── 统计信息更新
   ↓
6. 重复步骤5，直到接收到停止信号
```

**核心组件：**

```c
// 事件循环结构
struct aeEventLoop {
    int maxfd;                    // 当前最大的文件描述符
    int setsize;                  // 文件描述符数组大小
    long long timeEventNextId;    // 下一个时间事件ID
    aeFileEvent *events;          // 文件事件数组
    aeFiredEvent *fired;          // 已触发事件数组
    aeTimeEvent *timeEventHead;   // 时间事件链表头
    int stop;                     // 停止标志
    void *apidata;                // 平台特定数据（epoll/kqueue等）
};
```

### Redis 6.0的多线程I/O模型

Redis 6.0引入多线程I/O处理，但命令执行仍是单线程：

**配置参数：**
```conf
# 开启多线程I/O
io-threads-do-reads yes

# I/O线程数（建议设置为CPU核心数）
io-threads 4
```

**执行流程：**

```
客户端连接
    ↓
主线程接收连接
    ↓
读取客户端数据（多线程并行）
    ↓
主线程解析命令
    ↓
主线程执行命令（单线程，原子性）
    ↓
生成响应（多线程并行）
    ↓
发送响应给客户端（多线程并行）
```

**多线程实现细节：**

1. **读操作并行化：**
   - 主线程将待读取的客户端分配给I/O线程
   - 多个I/O线程并行从socket读取数据
   - 主线程等待所有I/O线程完成
   - 主线程按顺序解析命令

2. **写操作并行化：**
   - 主线程执行完命令后，将响应数据放入客户端输出缓冲区
   - 主线程将待发送的客户端分配给I/O线程
   - 多个I/O线程并行向socket写入数据
   - 主线程等待所有I/O线程完成

**线程安全保证：**

- 命令执行阶段仍然是单线程，保证原子性
- I/O线程只负责网络数据的读写，不访问共享数据结构
- 通过锁机制协调主线程和I/O线程的工作

### 为什么命令执行保持单线程？

1. **复杂性**：多线程访问共享数据需要复杂的锁机制，容易出错
2. **性能**：Redis的瓶颈不在CPU，多线程执行命令收益有限
3. **原子性**：单线程天然保证命令原子性，简化编程模型
4. **历史包袱**：改为多线程需要大量重构，风险高

### 多线程最佳实践

```
适用场景：
- Redis实例较大（内存>10GB）
- 网络带宽充足
- 有大量并发连接（>1000）
- 主要瓶颈在网络I/O

不适用场景：
- 执行复杂命令（如大key操作）
- CPU密集型操作
- 小实例，连接数少
- 内存是瓶颈
```

## 5. Redis有哪些使用场景？

**解答：**

Redis凭借其高性能和丰富的数据结构，在多个领域有广泛应用：

### 1. 缓存

**应用场景：**
- 页面缓存
- 对象缓存
- 查询结果缓存
- 会话缓存（Session）

**实现方式：**
```
读取数据：
1. 先查Redis缓存
2. 如果命中，直接返回
3. 如果未命中，查数据库
4. 将数据写入Redis，设置过期时间
5. 返回数据

写入数据：
1. 先更新数据库
2. 删除或更新Redis缓存
```

**缓存策略：**
- Cache-Aside（旁路缓存）
- Read-Through（读穿透）
- Write-Through（写穿透）
- Write-Behind（异步写）

### 2. 计数器

**应用场景：**
- 网站访问量统计
- 用户点赞数
- 视频播放量
- 接口限流计数

**实现方式：**
```bash
# 增加计数
INCR page_view:home

# 增加指定数量
INCRBY user:123:likes 1

# 获取当前值
GET page_view:home
```

**高级计数器 - 使用HyperLogLog：**
```bash
# UV统计（去重计数）
PFADD uv:20231201 user1 user2 user3
PFCOUNT uv:20231201
```

### 3. 分布式锁

**应用场景：**
- 防止重复提交
- 订单防重
- 库存扣减
- 定时任务互斥执行

**基础实现：**
```bash
# 加锁（NX：不存在时才设置，PX：设置过期时间）
SET lock:order:123 "owner_id" NX PX 30000

# 释放锁（使用Lua脚本保证原子性）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**Redisson框架实现：**
- 自动续期（看门狗机制）
- 可重入锁
- 公平锁
- 红锁（RedLock）多主节点锁

### 4. 会话存储（Session）

**应用场景：**
- 分布式系统会话共享
- 单点登录（SSO）
- 用户状态存储

**实现方式：**
```bash
# 存储用户会话
HSET session:user123 user_id 123 username "张三" login_time "2023-12-01 10:00:00"

# 设置过期时间（30分钟）
EXPIRE session:user123 1800

# 延长会话时间
EXPIRE session:user123 1800
```

### 5. 排行榜

**应用场景：**
- 游戏排行榜
- 热门文章排行
- 商品销量排行
- 积分排行

**实现方式：**
```bash
# 添加分数（Sorted Set）
ZADD leaderboard 1000 "player1"
ZADD leaderboard 1500 "player2"
ZADD leaderboard 800 "player3"

# 获取前10名（分数从高到低）
ZREVRANGE leaderboard 0 9 WITHSCORES

# 获取玩家排名
ZREVRANK leaderboard "player1"

# 增加分数
ZINCRBY leaderboard 100 "player1"
```

### 6. 消息队列

**应用场景：**
- 异步处理
- 任务队列
- 事件通知
- 延时任务

**List实现消息队列：**
```bash
# 生产者
LPUSH queue:email "{"to":"user@example.com","subject":"Hello"}"

# 消费者（阻塞读取）
BRPOP queue:email 0
```

**Pub/Sub发布订阅：**
```bash
# 订阅频道
SUBSCRIBE news.sports

# 发布消息
PUBLISH news.sports "NBA总决赛开始啦"
```

**Stream（Redis 5.0+）实现消息队列：**
```bash
# 添加消息
XADD mystream * field1 value1 field2 value2

# 消费者组
XGROUP CREATE mystream mygroup $
XREADGROUP GROUP mygroup consumer1 COUNT 1 STREAMS mystream >
```

### 7. 实时系统

**应用场景：**
- 实时统计
- 在线用户列表
- 实时推送

**Bitmap实现日活跃用户统计：**
```bash
# 设置用户今天登录（user_id为1000）
SETBIT dau:2023-12-01 1000 1

# 统计今天登录用户数
BITCOUNT dau:2023-12-01

# 统计连续两天都登录的用户
BITOP AND both_days dau:2023-12-01 dau:2023-11-30
BITCOUNT both_days
```

### 8. 地理位置应用

**应用场景：**
- 附近的人
- 店铺位置搜索
- 配送范围计算

**实现方式：**
```bash
# 添加位置
GEOADD locations 116.4074 39.9042 "北京"
GEOADD locations 121.4737 31.2304 "上海"

# 获取位置
GEOPOS locations "北京"

# 计算距离
GEODIST locations "北京" "上海" km

# 附近搜索（半径10公里）
GEORADIUS locations 116.4074 39.9042 10 km WITHDIST WITHCOORD
```

### 9. 限流

**应用场景：**
- API接口限流
- 防止刷票
- 防止暴力破解

**固定窗口限流：**
```bash
# 每分钟最多100次请求
key = "rate_limit:user123"
current = GET key
if current == null:
    SET key 1 EX 60
    return true
elif current < 100:
    INCR key
    return true
else:
    return false
```

**滑动窗口限流（使用Sorted Set）：**
```bash
# 每分钟最多100次请求
ZADD rate_limit:user123 <current_timestamp> <current_timestamp>
ZREMRANGEBYSCORE rate_limit:user123 0 <current_timestamp - 60>
count = ZCARD rate_limit:user123
if count <= 100:
    return true
else:
    ZREM rate_limit:user123 <current_timestamp>
    return false
```

### 10. 购物车

**应用场景：**
- 电商购物车
- 临时商品存储

**实现方式：**
```bash
# 添加商品到购物车
HSET cart:user123 product:1001 "{\"name\":\"iPhone\",\"price\":5999,\"quantity\":1}"

# 修改数量
HSET cart:user123 product:1001 "{\"name\":\"iPhone\",\"price\":5999,\"quantity\":2}"

# 获取购物车所有商品
HGETALL cart:user123

# 删除商品
HDEL cart:user123 product:1001

# 清空购物车
DEL cart:user123
```

## 6. Redis键的过期策略是什么？

**解答：**

Redis提供了键过期功能，可以通过`EXPIRE`、`PEXPIRE`、`EXPIREAT`、`PEXPIREAT`等命令设置键的过期时间。过期策略是指Redis如何处理已过期的键。

### 三种过期删除策略

Redis采用**惰性删除 + 定期删除**的组合策略：

#### 1. 惰性删除（Lazy Expiration）

**原理：**
当访问一个键时，Redis会先检查该键是否过期，如果过期则删除，否则返回该键的值。

**实现位置：**
```c
// db.c 中的 expireIfNeeded 函数
int expireIfNeeded(redisDb *db, robj *key) {
    // 获取键的过期时间
    mstime_t when = getExpire(db,key);
    
    // 如果没有设置过期时间，返回0
    if (when < 0) return 0;
    
    // 如果未过期，返回0
    if (mstime() <= when) return 0;
    
    // 已过期，删除键
    // 如果是主节点，删除并通知从节点
    // 如果是从节点，不删除（等待主节点同步删除命令）
    if (server.masterhost != NULL) return 1;
    
    // 删除过期键
    server.stat_expiredkeys++;
    propagateExpire(db,key,server.lazyfree_lazy_expire);
    notifyKeyspaceEvent(NOTIFY_EXPIRED,"expired",key,db->id);
    
    // 使用同步或异步方式删除
    return server.lazyfree_lazy_expire ? dbAsyncDelete(db,key) :
                                         dbSyncDelete(db,key);
}
```

**优点：**
- CPU友好：只在访问时才检查，不消耗额外CPU资源
- 内存友好：不会扫描所有键

**缺点：**
- 内存不友好：已过期的键如果不被访问，会一直占用内存
- 可能导致内存泄漏：大量过期键不被访问时会浪费内存

#### 2. 定期删除（Active Expiration）

**原理：**
Redis每隔一段时间（默认100ms）随机抽取一些键进行检查，删除其中已过期的键。

**实现细节：**

```c
// server.c 中的 activeExpireCycle 函数
void activeExpireCycle(int type) {
    // 每次检查的数据库数量
    unsigned int dbs_per_call = CRON_DBS_PER_CALL;
    
    // 每个数据库检查的键数量上限
    static unsigned int current_db = 0;
    int j;
    
    for (j = 0; j < dbs_per_call && timelimit_exit == 0; j++) {
        redisDb *db = server.db+(current_db % server.dbnum);
        current_db++;
        
        // 随机抽取20个键进行检查
        for (i = 0; i < 20; i++) {
            // 获取随机键
            de = dictGetRandomKey(db->expires);
            if (de == NULL) break;
            
            e = dictGetEntryVal(de);
            t = (long long) e;
            
            // 如果已过期，删除
            if (now > t) {
                sds key = dictGetEntryKey(de);
                robj *keyobj = createStringObject(key,sdslen(key));
                
                // 传播删除操作到AOF和从节点
                propagateExpire(db,keyobj,server.lazyfree_lazy_expire);
                
                // 删除键
                if (server.lazyfree_lazy_expire)
                    dbAsyncDelete(db,keyobj);
                else
                    dbSyncDelete(db,keyobj);
                
                decrRefCount(keyobj);
                server.stat_expiredkeys++;
            }
        }
        
        // 如果已过期键占比超过25%，继续检查
        if ((num / 20) > 0.25) {
            // 继续检查
        }
    }
}
```

**配置参数：**
```conf
# 定期删除的频率（每秒执行次数）
hz 10  # 默认10次/秒

# 动态调整hz（根据客户端连接数自动调整）
dynamic-hz yes
```

**优点：**
- 通过限制删除操作的时长和频率，减少对CPU的影响
- 有效释放过期键占用的内存

**缺点：**
- 难以确定删除操作的频率和时长
- 频率太高会消耗CPU，频率太低会浪费内存

#### 3. 定时删除（Redis不采用）

**原理：**
为每个设置过期时间的键创建一个定时器，当键过期时间到达时，立即删除。

**优点：**
- 内存友好：过期键会被立即删除

**缺点：**
- CPU不友好：需要大量定时器，占用CPU资源
- 实现复杂：需要高效的时间事件管理

**Redis不采用此策略。**

### 过期键对持久化的影响

#### RDB持久化

**生成RDB时：**
- 已过期的键不会被保存到RDB文件中

**加载RDB时：**
- 主服务器：只加载未过期的键，过期键被忽略
- 从服务器：加载所有键（包括过期键），但在数据同步完成后，从主服务器接收到DEL命令再删除

#### AOF持久化

**AOF写入：**
- 当键过期被删除时，会向AOF文件追加一条DEL命令

**AOF重写：**
- 已过期的键不会被写入重写后的AOF文件

#### 主从复制

**主服务器：**
- 键过期后，向所有从服务器发送DEL命令

**从服务器：**
- 接收到DEL命令后删除键
- 从服务器不会主动删除过期键（即使已过期也会返回给客户端）
- 保证主从数据一致性

### 内存淘汰策略

当Redis内存使用达到maxmemory限制且没有可删除的过期键时，需要执行内存淘汰策略：

```conf
# 配置最大内存
maxmemory 1gb

# 配置淘汰策略
maxmemory-policy allkeys-lru
```

**可选策略：**
- `noeviction`：不淘汰，返回错误（默认）
- `allkeys-lru`：在所有键中使用LRU算法淘汰
- `volatile-lru`：在设置了过期时间的键中使用LRU淘汰
- `allkeys-random`：在所有键中随机淘汰
- `volatile-random`：在设置了过期时间的键中随机淘汰
- `volatile-ttl`：淘汰即将过期的键（TTL小的优先）
- `allkeys-lfu`（Redis 4.0+）：使用LFU算法
- `volatile-lfu`（Redis 4.0+）：在设置了过期时间的键中使用LFU淘汰

## 7. 什么是Redis的大key问题？如何解决？

**解答：**

### 什么是大key

**定义：**
大key是指Redis中某个key对应的value占用大量内存或包含大量元素，通常有以下标准：

```
String类型：value大小 > 10KB
Hash/Set/ZSet/List类型：元素个数 > 5000个
```

**产生原因：**
1. 缓存了完整的序列化对象（如整个用户表）
2. 社交应用的粉丝列表、关注列表无限增长
3. 日志、消息队列数据累积
4. 统计类数据按时间维度累积（如每天的统计数据存储在一个key中）

### 大key的危害

**1. 内存不均：**
```
Redis Cluster场景：
- 某个节点存储了大量大key
- 导致该节点内存使用率远高于其他节点
- 集群整体内存利用率不均衡
```

**2. 超时阻塞：**
```
Redis单线程模型：
- 删除大key需要较长时间（DEL操作是同步的）
- 阻塞其他命令执行
- 客户端出现超时

示例：
- 删除包含100万个元素的List
- 可能需要几百毫秒甚至几秒
- 期间所有命令都无法执行
```

**3. 网络拥塞：**
```
- 读取大key会产生大量网络流量
- 占用网络带宽
- 影响其他操作

示例：
- 一个String key大小为10MB
- 100个并发读取就会产生1GB流量
```

**4. 持久化问题：**
```
RDB持久化：
- 生成RDB时，大key会占用大量内存
- 可能导致fork操作耗时过长
- 影响正常服务

AOF持久化：
- 大key修改会产生大量AOF日志
- AOF文件快速增长
- AOF重写耗时增加
```

**String类型拆分：**

拆分为多个小key：

## 8. Redis如何处理并发竞争问题？

**解答：**

并发竞争问题是指多个客户端同时操作同一个key，可能导致数据不一致或覆盖。Redis提供了多种机制来处理并发竞争问题。

### 1. 乐观锁（WATCH/MULTI/EXEC）

**原理：**
使用WATCH命令监视一个或多个key，如果在执行EXEC命令前，这些key被其他客户端修改，事务将被打断，返回nil。

**实现：**
```bash
# 客户端1
WATCH balance:user1
GET balance:user1    # 假设值为100
MULTI
DECRBY balance:user1 50
EXEC
# 如果在MULTI和EXEC之间，balance:user1被其他客户端修改
# EXEC将返回nil，事务失败

# 客户端2（在客户端1的WATCH和EXEC之间执行）
DECRBY balance:user1 30
```

**使用场景：**
- 余额扣减
- 库存扣减
- 需要原子性操作的场景

**Java代码示例：**
```java
public boolean deductBalance(String userId, int amount) {
    String key = "balance:" + userId;
    
    for (int i = 0; i < 3; i++) {  // 重试3次
        try {
            // 监视key
            redisTemplate.watch(key);
            
            // 获取当前值
            Integer balance = (Integer) redisTemplate.opsForValue().get(key);
            if (balance == null || balance < amount) {
                redisTemplate.unwatch();
                return false;
            }
            
            // 开启事务
            redisTemplate.multi();
            redisTemplate.opsForValue().set(key, balance - amount);
            
            // 执行事务
            List<Object> results = redisTemplate.exec();
            if (results != null && !results.isEmpty()) {
                return true;  // 执行成功
            }
            // 事务失败，重试
        } catch (Exception e) {
            redisTemplate.unwatch();
            throw e;
        }
    }
    return false;
}
```

**优点：**
- 不需要加锁，性能较好
- 适合读多写少的场景

**缺点：**
- 需要重试机制
- 冲突频繁时性能下降

### 2. 分布式锁

**基本原理：**
使用SETNX（Set if Not Exists）命令实现排他锁。

**基础实现：**
```bash
# 加锁
SET lock:order:123 owner_id NX EX 30

# 解锁（使用Lua脚本保证原子性）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**完整实现（含看门狗自动续期）：**

```java
@Component
public class RedisDistributedLock {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    private static final String LOCK_PREFIX = "lock:";
    private static final long DEFAULT_EXPIRE_TIME = 30;
    private static final String UNLOCK_SCRIPT = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "return redis.call('del', KEYS[1]) " +
        "else return 0 end";
    
    // 存储每个锁的续期任务
    private ConcurrentHashMap<String, ScheduledFuture<?>> renewalTasks = new ConcurrentHashMap<>();
    
    /**
     * 获取锁
     */
    public boolean tryLock(String lockKey, String requestId, long expireTime) {
        String key = LOCK_PREFIX + lockKey;
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(key, requestId, expireTime, TimeUnit.SECONDS);
        
        if (Boolean.TRUE.equals(success)) {
            // 启动看门狗自动续期
            startWatchDog(key, requestId, expireTime);
            return true;
        }
        return false;
    }
    
    /**
     * 看门狗自动续期
     */
    private void startWatchDog(String key, String requestId, long expireTime) {
        ScheduledExecutorService executor = Executors.newSingleThreadScheduledExecutor();
        ScheduledFuture<?> future = executor.scheduleAtFixedRate(() -> {
            try {
                // 检查锁是否仍由当前线程持有
                String currentValue = redisTemplate.opsForValue().get(key);
                if (requestId.equals(currentValue)) {
                    // 续期（重置过期时间）
                    redisTemplate.expire(key, expireTime, TimeUnit.SECONDS);
                }
            } catch (Exception e) {
                log.error("Lock renewal failed", e);
            }
        }, expireTime / 3, expireTime / 3, TimeUnit.SECONDS);
        
        renewalTasks.put(key, future);
    }
    
    /**
     * 释放锁
     */
    public boolean unlock(String lockKey, String requestId) {
        String key = LOCK_PREFIX + lockKey;
        
        try {
            // 执行Lua脚本原子性删除
            Long result = redisTemplate.execute(
                new DefaultRedisScript<>(UNLOCK_SCRIPT, Long.class),
                Collections.singletonList(key),
                requestId
            );
            
            // 取消续期任务
            ScheduledFuture<?> future = renewalTasks.remove(key);
            if (future != null) {
                future.cancel(false);
            }
            
            return result != null && result > 0;
        } catch (Exception e) {
            log.error("Unlock failed", e);
            return false;
        }
    }
}
```

**使用Redisson框架（推荐）：**

### 3. Lua脚本实现原子操作

**Java代码：**
```java
@Service
public class StockService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    private static final String DEDUCT_STOCK_SCRIPT = 
        "local stock = tonumber(redis.call('get', KEYS[1])); " +
        "if stock == nil then return -1 end; " +
        "if stock < tonumber(ARGV[1]) then return 0 end; " +
        "redis.call('decrby', KEYS[1], ARGV[1]); " +
        "return redis.call('get', KEYS[1]);";
    
    /**
     * 扣减库存
     * @return -1: key不存在, 0: 库存不足, >0: 剩余库存
     */
    public Long deductStock(String productId, int quantity) {
        String key = "stock:" + productId;
        
        DefaultRedisScript<Long> script = new DefaultRedisScript<>();
        script.setScriptText(DEDUCT_STOCK_SCRIPT);
        script.setResultType(Long.class);
        
        return redisTemplate.execute(script, Collections.singletonList(key), String.valueOf(quantity));
    }
}
```

**优点：**
- 真正的原子操作
- 减少网络往返次数
- 性能优于事务和锁

**缺点：**
- 脚本不能太大，否则影响其他命令执行
- 调试困难
- 逻辑复杂时难以维护

### 4. 利用Redis单线程特性

由于Redis是单线程执行命令，单个命令是原子的，可以利用这一点避免并发问题。

**原子性命令：**
```bash
# 计数器（原子性增减）
INCR counter
DECR counter
INCRBY counter 5

# 列表操作（原子性）
LPUSH list value
RPOP list
BLPOP list 10

# Hash操作（单个field原子性）
HINCRBY hash field 1
HSETNX hash field value

# Set操作
SADD set member
SPOP set

# Sorted Set操作
ZINCRBY zset 1 member
```

**示例：秒杀系统库存扣减**

```java
@Service
public class SeckillService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    /**
     * 使用decr原子性扣减库存
     */
    public boolean seckill(String productId, String userId) {
        String stockKey = "seckill:stock:" + productId;
        String orderKey = "seckill:orders:" + productId;
        
        // 1. 原子性扣减库存
        Long stock = redisTemplate.opsForValue().decrement(stockKey);
        
        if (stock == null || stock < 0) {
            // 库存不足，回补库存
            redisTemplate.opsForValue().increment(stockKey);
            return false;
        }
        
        // 2. 记录购买用户（使用set去重）
        Long added = redisTemplate.opsForSet().add(orderKey, userId);
        if (added == 0) {
            // 重复购买，回补库存
            redisTemplate.opsForValue().increment(stockKey);
            redisTemplate.opsForSet().remove(orderKey, userId);
            return false;
        }
        
        // 3. 发送异步消息处理订单
        // sendOrderMessage(productId, userId);
        
        return true;
    }
}
```

### 5. 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 乐观锁 | 无锁，性能好 | 冲突时需要重试 | 读多写少，冲突少 |
| 分布式锁 | 强一致性 | 性能较低，可能死锁 | 强一致性要求 |
| Lua脚本 | 真正的原子性，性能好 | 脚本不能太复杂 | 复杂原子操作 |
| 原子命令 | 最简单，性能最好 | 功能有限 | 简单计数、列表操作 |

### 6. 最佳实践

**1. 优先使用原子命令：**
```java
// 好 - 使用原子命令
redisTemplate.opsForValue().decrement("stock");

// 不好 - 需要加锁
Long stock = (Long) redisTemplate.opsForValue().get("stock");
redisTemplate.opsForValue().set("stock", stock - 1);
```

**2. Lua脚本优于事务：**
```java
// 推荐 - Lua脚本
String script = "redis.call('decr', KEYS[1]); redis.call('sadd', KEYS[2], ARGV[1]);";

// 不推荐 - 事务
redisTemplate.multi();
redisTemplate.opsForValue().decrement("stock");
redisTemplate.opsForSet().add("orders", userId);
redisTemplate.exec();
```

**3. 减少锁的粒度：**
```java
// 不好 - 锁整个库存
lock("lock:stock");

// 好 - 锁具体商品
lock("lock:stock:" + productId);
```

**4. 设置合理的锁超时时间：**
```java
// 必须设置超时时间，防止死锁
lock.tryLock(10, 30, TimeUnit.SECONDS);
```

**5. 锁必须可重入：**
```java
// 使用Redisson实现可重入锁
RLock lock = redissonClient.getLock("lock:order");
lock.lock();
try {
    // 业务逻辑
    // 可以再次获取同一把锁（可重入）
    lock.lock();
    try {
        // 嵌套业务逻辑
    } finally {
        lock.unlock();
    }
} finally {
    lock.unlock();
}
```

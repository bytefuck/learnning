# Redis面试题 - 持久化篇

## 1. Redis有哪些持久化方式？各自的优缺点是什么？

**解答：**

Redis提供了两种主要的持久化方式：RDB（Redis Database）和AOF（Append Only File）。Redis 4.0版本之后还引入了RDB-AOF混合持久化。

### RDB持久化

#### 原理

RDB通过创建数据集的快照来实现持久化。在指定的时间间隔内，将内存中的数据集快照写入磁盘。

**触发方式：**
```
1. 手动触发：
   - SAVE：阻塞主进程，直到RDB文件创建完成
   - BGSAVE：fork子进程，异步创建RDB文件

2. 自动触发：
   - 配置save参数，满足条件时自动执行BGSAVE
   - 执行FLUSHALL命令（如果配置了save）
   - 主从复制时，从节点同步会触发主节点生成RDB
   - 执行SHUTDOWN命令时
```

#### 配置参数

```conf
# save <seconds> <changes>
# 900秒内至少有1个key被修改，触发保存
save 900 1
# 300秒内至少有10个key被修改，触发保存
save 300 10
# 60秒内至少有10000个key被修改，触发保存
save 60 10000

# 当后台保存出错时，停止写入操作（防止数据不一致）
stop-writes-on-bgsave-error yes

# 是否压缩RDB文件
rdbcompression yes

# 是否校验RDB文件CRC64校验和
rdbchecksum yes

# RDB文件名
dbfilename dump.rdb

# RDB文件存放目录
dir /var/lib/redis
```

#### RDB文件结构

```
RDB文件二进制结构：

┌─────────────────────────────────────────────────────┐
│                    RDB Header                        │
│  "REDIS" + version + aux fields                     │
│  例如：REDIS0009                                     │
├─────────────────────────────────────────────────────┤
│                   Database 0                         │
│  FE 00 - 选择数据库0                                 │
├─────────────────────────────────────────────────────┤
│                    Key-Value Pairs                   │
│  FD + expire time (seconds)                          │
│  FC + expire time (milliseconds)                     │
│  value type + key + value                            │
│  例如：00 03 6b6579 05 76616c7565                    │
│        (type=string, key="key", value="value")       │
├─────────────────────────────────────────────────────┤
│                   Database 1...N                     │
├─────────────────────────────────────────────────────┤
│                   End of File                        │
│  FF + 8 bytes checksum                               │
└─────────────────────────────────────────────────────┘
```

#### 优缺点

**优点：**
1. **性能高**：BGSAVE由子进程执行，不阻塞主进程
2. **文件紧凑**：经过压缩，体积小，适合备份和传输
3. **恢复速度快**：加载RDB文件比重放AOF日志快得多
4. **数据安全性**：适合灾难恢复，可以定期备份RDB文件

**缺点：**
1. **可能丢失数据**：最后一次快照后的数据会丢失（分钟级）
2. **fork开销大**：数据集大时，fork可能耗时较久，阻塞主进程
3. **保存时内存占用高**：COW（写时复制）机制下，保存期间内存使用量翻倍

#### 适用场景

- 数据备份
- 灾难恢复
- 对数据完整性要求不高的场景
- 大数据集快速恢复

### AOF持久化

#### 原理

AOF以日志的形式记录每个写操作（增删改），将操作追加到文件末尾。重启时重新执行这些命令来恢复数据。

**同步策略：**
```conf
# 每次写入都同步（最安全，最慢）
appendfsync always

# 每秒同步一次（默认，平衡安全和性能）
appendfsync everysec

# 由操作系统决定何时同步（最快，最不安全）
appendfsync no
```

#### 配置参数

```conf
# 开启AOF
appendonly yes

# AOF文件名
appendfilename "appendonly.aof"

# 同步策略
appendfsync everysec

# AOF重写期间是否禁止fsync（防止重写时磁盘IO压力过大）
no-appendfsync-on-rewrite no

# 自动重写触发条件
# AOF文件大小超过上一次重写后大小的100%时触发
auto-aof-rewrite-percentage 100
# AOF文件最小大小（小于64MB不触发重写）
auto-aof-rewrite-min-size 64mb
```

#### AOF文件结构

```
AOF文件文本格式：

*2          # 命令参数个数
$6          # 第一个参数长度
SELECT      # 第一个参数
$1          # 第二个参数长度
0           # 第二个参数（选择数据库0）
*3          # 命令参数个数
$3          # 第一个参数长度
SET         # 第一个参数
$4          # 第二个参数长度
name        # 第二个参数
$6          # 第三个参数长度
Redis       # 第三个参数
*4          # 命令参数个数
$5          # 第一个参数长度
HSET        # 第一个参数
$4          # 第二个参数长度
user         # 第二个参数
$4          # 第三个参数长度
name         # 第三个参数
$5          # 第四个参数长度
Alice        # 第四个参数
```

#### AOF重写

**为什么需要重写：**
```
问题：AOF文件持续增长，包含大量冗余命令

示例：
SET key value1
SET key value2
SET key value3

最终key的值是value3，但AOF记录了3条命令
```

**重写原理：**
```
1. fork子进程
2. 子进程读取当前数据集，生成新的AOF文件（只包含当前值的命令）
3. 重写期间，父进程将新命令写入AOF重写缓冲区
4. 重写完成后，将缓冲区的命令追加到新AOF文件
5. 用新文件替换旧文件
```

**重写流程：**
```c
void rewriteAppendOnlyFileBackground(void) {
    pid_t childpid;
    
    if ((childpid = fork()) == 0) {
        // 子进程
        // 创建新的AOF文件
        rewriteAppendOnlyFile(tmpfile);
        _exit(0);
    } else {
        // 父进程
        // 记录重写开始时间
        server.aof_rewrite_time_start = time(NULL);
        // 标记正在重写
        server.aof_rewrite_scheduled = 1;
    }
}
```

#### 优缺点

**优点：**
1. **数据安全性高**：everysec策略最多丢失1秒数据
2. **可读性强**：AOF是文本格式，可以人工查看和编辑
3. **支持重写**：自动压缩AOF文件，避免无限增长

**缺点：**
1. **文件体积大**：通常比RDB文件大
2. **恢复速度慢**：需要重放所有命令
3. **性能开销**：每秒fsync对性能有一定影响
4. **重写时磁盘压力大**：重写期间磁盘IO可能达到峰值

#### 适用场景

- 对数据完整性要求高的场景
- 不能接受数据丢失的场景
- 需要查看操作日志的场景

### RDB-AOF混合持久化（Redis 4.0+）

#### 原理

混合持久化结合了RDB和AOF的优点：
- 重写AOF文件时，先将当前数据集以RDB格式写入文件开头
- 再将重写期间的增量命令以AOF格式追加到文件末尾

```
混合AOF文件结构：

┌─────────────────────────────────────────────────────┐
│                   RDB Format Data                    │
│  当前数据集的RDB快照（二进制）                        │
│  占用大部分空间                                      │
├─────────────────────────────────────────────────────┤
│                   AOF Format Data                    │
│  重写期间的增量命令（文本）                           │
│  占用小部分空间                                      │
└─────────────────────────────────────────────────────┘
```

#### 配置参数

```conf
# 开启混合持久化
aof-use-rdb-preamble yes
```

#### 优缺点

**优点：**
1. **快速恢复**：加载RDB部分快速恢复大部分数据
2. **数据完整**：AOF部分保证数据不丢失
3. **文件适中**：比纯AOF文件小，比RDB文件稍大

**缺点：**
1. **兼容性**：Redis 4.0之前的版本无法读取
2. **稍微复杂**：文件格式是混合的

### 持久化方式对比

| 特性 | RDB | AOF | RDB-AOF混合 |
|------|-----|-----|-------------|
| 文件体积 | 小（压缩） | 大 | 中等 |
| 恢复速度 | 快 | 慢 | 快 |
| 数据安全性 | 可能丢失几分钟 | 最多丢失1秒 | 最多丢失1秒 |
| 性能影响 | 低（fork时较高） | 中等（fsync） | 中等 |
| 可读性 | 二进制 | 文本 | 混合 |
| 重写机制 | 无 | 有 | 有 |
| 推荐场景 | 备份、容灾 | 高可用 | 生产环境 |

### 最佳实践

**1. 同时开启RDB和AOF（推荐）**

## 2. AOF重写是什么？为什么要重写？重写过程中数据会丢失吗？

**解答：**

### 什么是AOF重写

AOF重写（Rewrite）是指生成一个新的AOF文件，用最少量的命令来重建当前数据集，从而减小AOF文件体积。

**示例：**
```
原始AOF文件（包含冗余命令）：
SET counter 0
INCR counter
INCR counter
INCR counter
SET user:1:name "Alice"
HSET user:1 age 25
HSET user:1 city "Beijing"
DEL counter

重写后的AOF文件（只包含当前状态）：
HSET user:1 name "Alice" age 25 city "Beijing"

体积从100行减少到1行
```

### 为什么要重写

**1. 减少文件体积**
```
问题：AOF文件持续增长
- 每次写操作都追加到文件
- 同一个key被多次修改会产生多条记录
- 已删除的key的记录仍然存在

示例：
LPUSH list a
LPUSH list b
LPUSH list c
LPOP list
LPOP list

最终list中只有[c]，但AOF记录了5条命令
```

**2. 提高恢复速度**
```
小文件加载更快：
- 100MB的AOF文件可能需要10秒加载
- 重写后10MB的AOF文件只需要1秒
```

**3. 减少磁盘空间占用**
```
长期运行的Redis实例：
- AOF文件可能达到数十GB
- 重写后可能只有几百MB
```

### 重写原理和流程

```
重写流程：

1. fork子进程
   父进程 ──fork──→ 子进程
   
2. 子进程扫描数据集
   子进程读取当前所有key-value
   生成新的AOF文件（最小命令集）
   
3. 父进程继续服务
   同时将新命令写入：
   - 原AOF文件（继续持久化）
   - AOF重写缓冲区（内存）
   
4. 子进程完成重写
   子进程退出
   
5. 父进程合并
   将重写缓冲区的命令追加到新AOF文件
   原子性替换旧AOF文件
```

**详细步骤：**

```c
// aof.c 中的重写函数
int rewriteAppendOnlyFile(char *filename) {
    // 1. 创建临时文件
    rio aof;
    rioInitWithFile(&aof, fp);
    
    // 2. 写入SELECT命令（选择数据库）
    for (j = 0; j < server.dbnum; j++) {
        // 跳过空数据库
        if (dictSize(server.db[j].dict) == 0) continue;
        
        // 写入数据库编号
        rioWrite(&aof, selectcmd, sizeof(selectcmd)-1);
        
        // 3. 遍历所有key
        dictIterator *di = dictGetIterator(server.db[j].dict);
        dictEntry *de;
        
        while ((de = dictNext(di)) != NULL) {
            sds keystr = dictGetKey(de);
            robj key, *o;
            
            // 根据value类型生成对应命令
            if (o->type == OBJ_STRING) {
                // 生成SET命令
                rewriteStringObject(&aof, &key, o);
            } else if (o->type == OBJ_LIST) {
                // 生成RPUSH命令
                rewriteListObject(&aof, &key, o);
            } else if (o->type == OBJ_SET) {
                // 生成SADD命令
                rewriteSetObject(&aof, &key, o);
            } else if (o->type == OBJ_ZSET) {
                // 生成ZADD命令
                rewriteSortedSetObject(&aof, &key, o);
            } else if (o->type == OBJ_HASH) {
                // 生成HSET命令
                rewriteHashObject(&aof, &key, o);
            }
        }
        dictReleaseIterator(di);
    }
    
    // 4. 刷新到磁盘
    fflush(fp);
    fsync(fileno(fp));
    fclose(fp);
    
    // 5. 原子性重命名文件
    rename(tmpfile, filename);
}
```

**重写缓冲区（AOF Rewrite Buffer）：**

```c
// 重写期间，新命令同时写入原AOF和重写缓冲区
void feedAppendOnlyFile(struct redisCommand *cmd, int dictid, robj **argv, int argc) {
    sds buf = sdscatprintf(sdsempty(), "*%d\r\n", argc);
    
    // 构建AOF协议格式的命令
    for (j = 0; j < argc; j++) {
        // ...
    }
    
    // 写入原AOF文件
    if (server.aof_state == AOF_ON)
        server.aof_buf = sdscatlen(server.aof_buf, buf, sdslen(buf));
    
    // 如果在重写期间，同时写入重写缓冲区
    if (server.aof_child_pid != -1)
        aofRewriteBufferAppend((unsigned char*)buf, sdslen(buf));
}
```

### 重写过程中数据会丢失吗？

**答案：不会丢失**

**原因：**

1. **fork时的数据不会丢失**
   - fork采用COW（写时复制）机制
   - 子进程复制父进程的内存页表
   - 父进程修改数据时，会复制被修改的页
   - 子进程看到的是fork时的数据快照

2. **重写期间的新数据不会丢失**
   - 父进程将新命令同时写入：
     - 原AOF文件（保证当前AOF完整）
     - 重写缓冲区（内存）
   - 重写完成后，将缓冲区的命令追加到新AOF文件

3. **原子性替换**
   - 使用rename系统调用原子性替换文件
   - 不会出现AOF文件不完整的情况

**可能的问题：**

```
1. 重写期间内存使用增加
   - 重写缓冲区占用内存
   - COW机制导致内存页复制
   - 极端情况下可能OOM

2. 磁盘IO压力
   - 重写期间磁盘写入量大
   - 可能影响Redis性能

3. 重写失败
   - 如果子进程崩溃，会保留原AOF文件
   - 不会丢失数据，但重写失败
```

### 重写触发条件

**自动触发：**

```bash
# 查看重写状态

```

**1. 调整触发阈值**


## 3. Redis重启时如何加载数据？加载顺序是什么？

**解答：**

### 数据加载顺序

Redis重启时，数据加载遵循以下优先级：

```
1. 检查是否开启AOF（appendonly yes）
   ↓ 是
2. 检查AOF文件是否存在
   ↓ 是
3. 加载AOF文件（忽略RDB）
   ↓ 否
4. 检查RDB文件是否存在
   ↓ 是
5. 加载RDB文件
   ↓ 否
6. 启动空实例
```

**代码逻辑：**
```c
// server.c 中的加载逻辑
void loadDataFromDisk(void) {
    long long start = ustime();
    
    // 1. 如果开启了AOF，优先加载AOF
    if (server.aof_state == AOF_ON) {
        if (loadAppendOnlyFile(server.aof_filename) == C_OK)
            serverLog(LL_NOTICE, "DB loaded from append only file: %.3f seconds",
                (float)(ustime()-start)/1000000);
    } else {
        // 2. 否则加载RDB
        if (rdbLoad(server.rdb_filename, &rsi) == C_OK) {
            serverLog(LL_NOTICE, "DB loaded from disk: %.3f seconds",
                (float)(ustime()-start)/1000000);
        } else if (errno != ENOENT) {
            serverLog(LL_WARNING, "Fatal error loading the DB: %s. Exiting.",
                strerror(errno));
            exit(1);
        }
    }
}
```

### AOF加载过程

**1. 检查AOF文件格式**
```c
int loadAppendOnlyFile(char *filename) {
    FILE *fp = fopen(filename, "r");
    
    // 检查是否是RDB-preamble格式（混合持久化）
    char sig[5];
    if (fread(sig, 1, 5, fp) != 5) {
        // 文件为空或读取失败
        return C_ERR;
    }
    
    if (memcmp(sig, "REDIS", 5) == 0) {
        // 混合格式：先加载RDB部分
        serverLog(LL_NOTICE, "Reading RDB preamble from AOF file...");
        // 回退5字节，从开头加载RDB
        fseek(fp, 0, SEEK_SET);
        if (rdbLoadRio(&rdb, NULL, 0, &rsi) != C_OK) {
            // RDB加载失败
            return C_ERR;
        }
        serverLog(LL_NOTICE, "RDB preamble of AOF file is loaded.");
    } else {
        // 纯AOF格式，回退到开头
        fseek(fp, 0, SEEK_SET);
    }
    
    // 继续加载AOF部分...
}
```

**2. 逐条执行AOF命令**
```c
// 读取并执行AOF文件中的命令
while (fgets(buf, sizeof(buf), fp) != NULL) {
    // 解析命令参数个数（*3）
    if (buf[0] != '*') goto fmterr;
    argc = atoi(buf+1);
    
    // 读取每个参数
    argv = zmalloc(sizeof(robj*) * argc);
    for (j = 0; j < argc; j++) {
        // 读取参数长度（$4）
        if (fgets(buf, sizeof(buf), fp) == NULL) goto fmterr;
        len = atoi(buf+1);
        
        // 读取参数内容
        argsds = sdsnewlen(NULL, len);
        if (fread(argsds, len, 1, fp) == 0) goto fmterr;
        argv[j] = createObject(OBJ_STRING, argsds);
        
        // 读取\r\n
        if (fread(buf, 2, 1, fp) == 0) goto fmterr;
    }
    
    // 查找命令
    cmd = lookupCommand(argv[0]->ptr);
    if (!cmd) {
        serverLog(LL_WARNING, "Unknown command '%s' reading the AOF file");
        exit(1);
    }
    
    // 执行命令
    cmd->proc(c);
    
    // 释放参数
    for (j = 0; j < argc; j++) decrRefCount(argv[j]);
    zfree(argv);
}
```

**3. 处理AOF截断**
```conf
# 如果AOF文件末尾不完整（如断电），是否加载截断后的数据
aof-load-truncated yes

# yes: 加载并记录警告
# no: 启动失败
```

### RDB加载过程

**1. 读取RDB头部**
```c
int rdbLoadRio(rio *rdb, int flags, rdbSaveInfo *rsi) {
    char buf[1024];
    
    // 读取魔术字符串"REDIS"
    if (rioRead(rdb, buf, 9) == 0) goto eoferr;
    buf[9] = '\0';
    if (memcmp(buf, "REDIS", 5) != 0) {
        serverLog(LL_WARNING, "Wrong signature trying to load DB from file");
        return C_ERR;
    }
    
    // 读取版本号
    rdbver = atoi(buf+5);
    if (rdbver < 1 || rdbver > RDB_VERSION) {
        serverLog(LL_WARNING, "Can't handle RDB format version %d", rdbver);
        return C_ERR;
    }
    
    // 读取辅助字段...
}
```

**2. 读取数据库数据**
```c
// 循环读取直到遇到结束标记FF
while ((type = rdbLoadType(rdb)) != -1) {
    if (type == RDB_OPCODE_SELECTDB) {
        // 选择数据库
        dbid = rdbLoadLen(rdb, NULL);
        if (dbid < 0 || dbid >= server.dbnum) {
            serverLog(LL_WARNING, "FATAL: Data file was created with a Redis ");
            return C_ERR;
        }
        db = server.db + dbid;
        continue;
    }
    
    if (type == RDB_OPCODE_EOF) {
        // 结束标记
        break;
    }
    
    // 读取过期时间
    if (type == RDB_OPCODE_EXPIRETIME_MS) {
        expiretime = rdbLoadMillisecondTime(rdb, rdbver);
        type = rdbLoadType(rdb);
    }
    
    // 读取key
    key = rdbLoadStringObject(rdb);
    
    // 根据类型读取value
    if (type == RDB_TYPE_STRING) {
        val = rdbLoadStringObject(rdb);
    } else if (type == RDB_TYPE_LIST) {
        val = rdbLoadListObject(rdb);
    } else if (type == RDB_TYPE_SET) {
        val = rdbLoadSetObject(rdb);
    } else if (type == RDB_TYPE_ZSET) {
        val = rdbLoadZSetObject(rdb);
    } else if (type == RDB_TYPE_HASH) {
        val = rdbLoadHashObject(rdb);
    }
    
    // 添加到数据库
    dbAdd(db, key, val);
    
    // 设置过期时间
    if (expiretime != -1) {
        setExpire(db, key, expiretime);
    }
}
```

### 常见问题

**Q1: 同时有AOF和RDB文件，加载哪个？**
```
优先加载AOF。
原因：AOF数据更完整（如果配置了everysec）。
```

```bash
# 1. 备份原文件
```


## 4. 如何在不影响业务的情况下迁移Redis数据？

**解答：**

### 迁移方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| RDB迁移 | 简单，快速 | 需要停止写入 | 可以停机的场景 |
| AOF迁移 | 完整 | 文件大，恢复慢 | 小数据量 |
| 主从同步 | 不停机，实时 | 配置复杂 | 大数据量 |
| redis-shake | 专业工具 | 需要额外部署 | 复杂迁移 |
| 双写 | 零停机 | 代码侵入 | 关键业务 |

### 方案1：RDB文件迁移

**适用场景：**
- 可以接受短暂停机
- 数据量适中
- 不需要实时同步

**步骤：**
```bash
# 1. 停止写入（或切换到只读）
redis-cli CONFIG SET appendonly no

# 2. 执行bgsave生成最新RDB
redis-cli BGSAVE

# 3. 等待保存完成
redis-cli INFO persistence | grep rdb_bgsave_in_progress
# 等待返回0

# 4. 复制RDB文件到新服务器
scp /var/lib/redis/dump.rdb new-server:/var/lib/redis/

# 5. 启动新Redis实例
redis-server /etc/redis/redis.conf

# 6. 验证数据完整性
redis-cli INFO keyspace
redis-cli DBSIZE
```

**优化：不停机迁移**
```bash
# 使用主从复制思路，但只同步一次

# 1. 新服务器作为从节点连接旧服务器
redis-cli --cluster replicate old-server:6379

# 2. 等待同步完成
redis-cli INFO replication
# master_sync_in_progress:0

# 3. 提升新服务器为主节点
redis-cli SLAVEOF NO ONE

# 4. 切换应用连接到新服务器
```

### 方案2：主从同步迁移

**适用场景：**
- 不能停机
- 需要实时同步
- 大数据量

**步骤：**
```bash
# 旧服务器（主节点）
# 1. 确认配置
redis-cli CONFIG GET bind
redis-cli CONFIG GET protected-mode

# 新服务器（从节点）
# 2. 配置从节点
redis-cli SLAVEOF old-server-ip 6379

# 3. 查看同步状态
redis-cli INFO replication
# 输出：
# role:slave
# master_host:old-server-ip
# master_link_status:up
# master_last_io_seconds_ago:1
# master_sync_in_progress:0

# 4. 等待同步完成（根据数据量）
# 可以实时监控
watch -n 1 'redis-cli INFO replication | grep master_last_io_seconds_ago'

# 5. 同步完成后，提升为主节点
redis-cli SLAVEOF NO ONE

# 6. 可选：让旧服务器成为新主节点的从节点（回滚用）
# redis-cli SLAVEOF new-server-ip 6379
```

**注意事项：**
```conf
# 1. 确保maxmemory足够大
# 同步期间可能有额外的内存开销

# 2. 调整repl-backlog-size
# 避免同步过程中断开导致全量同步
repl-backlog-size 256mb

# 3. 监控同步延迟
redis-cli INFO replication | grep master_last_io_seconds_ago
# 如果这个值持续增长，说明有延迟
```

### 方案3：redis-shake工具迁移

**介绍：**
redis-shake是阿里云开源的Redis数据迁移工具，支持：
- 全量迁移
- 增量同步
- 双向同步
- 数据过滤

**安装：**
```bash
# 下载
wget https://github.com/alibaba/RedisShake/releases/download/release-v2.0.3-20201218/redis-shake-v2.0.3.tar.gz

# 解压
tar -xzf redis-shake-v2.0.3.tar.gz
cd redis-shake-v2.0.3
```

**配置（sync模式）：**
```toml
# redis-shake.conf

# 源Redis
source.type = standalone
source.address = old-server:6379
source.password = old-password
source.auth_type = auth

# 目标Redis
target.type = standalone
target.address = new-server:6379
target.password = new-password
target.auth_type = auth

# 模式：sync（全量+增量）、rump（仅全量）
mode = sync
```

**执行迁移：**
```bash
# 启动迁移
./redis-shake -type sync -conf redis-shake.conf

# 输出：
# 2023/01/01 10:00:00 [INFO] sync start...
# 2023/01/01 10:00:05 [INFO] full sync done, offset: 123456
# 2023/01/01 10:00:05 [INFO] incremental sync start...

# 等待显示增量同步正常后，可以切换应用
```

**切换流程：**
```bash
# 1. 停止写入（或进入只读模式）
redis-cli CONFIG SET save ""  # 禁止旧服务器自动保存

# 2. 等待redis-shake同步追上
# 观察日志，确保master_offset - slave_offset < 1000

# 3. 停止redis-shake
Ctrl+C

# 4. 切换应用到新服务器
# 修改应用配置，连接到new-server

# 5. 验证
redis-cli -h new-server INFO keyspace
```

### 方案4：双写迁移（零停机）

**适用场景：**
- 关键业务，绝对不能停机
- 有代码修改权限

**架构：**
```
应用层双写：
┌─────────────────────────────────────┐
│           Application               │
│                                     │
│   writeToOldRedis(data);            │
│   writeToNewRedis(data);  // 新增   │
│                                     │
└──────────────┬──────────────────────┘
               │
      ┌────────┴────────┐
      ↓                 ↓
┌──────────┐      ┌──────────┐
│ Old Redis│      │ New Redis│
│ (主)     │      │ (从)     │
└──────────┘      └──────────┘
```

**步骤：**

**阶段1：双写（2周）**
```java
@Service
public class RedisMigrationService {
    
    @Autowired
    private StringRedisTemplate oldRedisTemplate;
    
    @Autowired
    private StringRedisTemplate newRedisTemplate;  // 新增
    
    public void set(String key, String value) {
        // 先写旧Redis
        oldRedisTemplate.opsForValue().set(key, value);
        
        // 再写新Redis（异步，不影响主流程）
        CompletableFuture.runAsync(() -> {
            try {
                newRedisTemplate.opsForValue().set(key, value);
            } catch (Exception e) {
                log.error("写入新Redis失败", e);
                // 可以发送到消息队列，稍后补偿
            }
        });
    }
}
```

**阶段2：数据补齐**
```bash
# 使用redis-shake同步历史数据
./redis-shake -type rump -conf redis-shake.conf

# 或使用scan命令批量同步
redis-cli --scan --pattern '*' | while read key; do
    redis-cli -h new-server MIGRATE new-server 6379 "" 0 5000 COPY KEYS "$key"
done
```

**阶段3：切换读取**
```java
public String get(String key) {
    // 先查新Redis
    String value = newRedisTemplate.opsForValue().get(key);
    if (value != null) {
        return value;
    }
    
    // 未命中，查旧Redis
    value = oldRedisTemplate.opsForValue().get(key);
    if (value != null) {
        // 回填新Redis
        newRedisTemplate.opsForValue().set(key, value);
    }
    
    return value;
}
```

**阶段4：完全切换**
```java
// 停止写入旧Redis
public void set(String key, String value) {
    // oldRedisTemplate.opsForValue().set(key, value);  // 注释掉
    newRedisTemplate.opsForValue().set(key, value);
}
```

**阶段5：下线旧Redis**
```bash
# 确认新Redis运行稳定后
# 1. 停止双写代码
# 2. 备份旧Redis数据
# 3. 下线旧Redis实例
```

## 5. 持久化过程中如何保证数据不丢失？

**解答：**

### 数据丢失场景分析

| 场景 | 可能丢失的数据 | 概率 |
|------|---------------|------|
| Redis崩溃 | RDB：上次快照后的数据<br>AOF：上次fsync后的数据 | 取决于配置 |
| 系统崩溃 | RDB：上次快照后的数据<br>AOF：取决于appendfsync | 取决于配置 |
| 磁盘损坏 | 所有未备份的数据 | 低 |
| 误操作 | 被删除的数据 | 人为 |

### RDB持久化防丢失

**1. 缩短快照间隔**
```conf
# 更频繁的保存
save 60 1000    # 1分钟内1000次修改就保存
save 30 100     # 30秒内100次修改就保存

# 问题：频繁fork影响性能
```

**2. 手动触发保存**
```bash
# 关键操作后手动保存
redis-cli BGSAVE

# 等待完成
while [ $(redis-cli INFO persistence | grep rdb_bgsave_in_progress | cut -d: -f2) -eq 1 ]; do
    sleep 1
done
```

**3. 备份RDB文件**
```bash
#!/bin/bash
# 定时备份脚本

# 1. 触发bgsave
redis-cli BGSAVE

# 2. 等待完成
sleep 10

# 3. 备份到远程
scp /var/lib/redis/dump.rdb backup-server:/backup/redis/$(date +%Y%m%d_%H%M%S).rdb

# 4. 保留最近30个备份
ls -t /backup/redis/*.rdb | tail -n +31 | xargs rm -f
```

### AOF持久化防丢失

**1. 选择合适的同步策略**
```conf
# always：最安全，性能最差
# 每次写入都fsync，几乎不会丢失数据
appendfsync always

# everysec：平衡（推荐）
# 每秒fsync一次，最多丢失1秒数据
appendfsync everysec

# no：最快，最不安全
# 由操作系统决定，可能丢失30秒数据
appendfsync no
```

**2. 多副本策略**
```conf
# 开启AOF和RDB，互为备份
appendonly yes
save 900 1
save 300 10

# 即使AOF损坏，还有RDB
```

**3. AOF重写期间的防护**
```conf
# 重写期间禁止fsync（避免磁盘IO压力）
no-appendfsync-on-rewrite yes

# 但这样重写期间的数据依赖于操作系统的fsync
# 如果系统崩溃，可能丢失重写期间的数据

# 如果数据安全优先，设置为no
no-appendfsync-on-rewrite no
```

### 综合防护策略

**1. 3-2-1备份原则**
```
- 3份数据副本
- 2种不同存储介质
- 1份异地备份

示例：
- 主节点：实时数据
- 从节点：实时同步（1份副本）
- RDB备份：本地磁盘（2种介质）
- RDB远程备份：云存储（异地）
```

**2. 主从复制 + 持久化**
```conf
# 主节点配置
appendonly yes
appendfsync everysec
save 900 1

# 从节点配置
# 从节点也开启持久化，作为额外备份
appendonly yes
appendfsync everysec
save 900 1
```

**3. 监控和告警**
```bash
#!/bin/bash
# 监控脚本

# 检查AOF最后写入时间
last_write=$(redis-cli INFO persistence | grep aof_last_write_status)
if [[ $last_write != *ok* ]]; then
    echo "AOF写入异常: $last_write" | mail -s "Redis告警" admin@example.com
fi

# 检查RDB状态
rdb_status=$(redis-cli INFO persistence | grep rdb_last_bgsave_status)
if [[ $rdb_status != *ok* ]]; then
    echo "RDB保存异常: $rdb_status" | mail -s "Redis告警" admin@example.com
fi

# 检查复制延迟
if [ $(redis-cli INFO replication | grep role | grep slave | wc -l) -gt 0 ]; then
    delay=$(redis-cli INFO replication | grep master_last_io_seconds_ago | cut -d: -f2)
    if [ $delay -gt 10 ]; then
        echo "复制延迟: ${delay}秒" | mail -s "Redis告警" admin@example.com
    fi
fi
```

# G1 垃圾回收器详解

## 一、G1 概述

**G1 (Garbage First)** 是 JDK 9+ 的默认垃圾收集器，面向服务端应用，目标是：
- 可预测的停顿时间模型
- 高吞吐量与低延迟的平衡
- 支持大内存（4GB~32GB+）

---

## 二、核心设计

### 1. Region 分区设计

```
┌─────────────────────────────────────────────────────────┐
│                      G1 堆内存布局                       │
├─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────────┤
│ E   │ E   │ S   │ S   │ O   │ O   │ O   │ H   │   ...   │
└─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────────┘
  ↓     ↓     ↓     ↓     ↓           ↓     ↓
 Eden  Eden  Surv  Surv  Old        Old  Humongous

E = Eden    S = Survivor    O = Old    H = Humongous（大对象）
```

**特点：**
- 堆被划分为多个大小相等的 Region（默认 2048 个）
- 每个 Region 大小 = 堆大小 / 2048（1MB~32MB，2的幂次）
- Region 动态角色：可充当 Eden、Survivor、Old
- 大对象（>Region 50%）直接分配到 Humongous Region

### 2. 关键数据结构

```
┌─────────────────────────────────────────────────────┐
│                    RSet (Remembered Set)             │
│  每个Region维护，记录"谁引用了我"（跨Region引用）     │
│  避免全堆扫描，实现精准的回收范围                     │
├─────────────────────────────────────────────────────┤
│                    CSet (Collection Set)            │
│  每次GC要回收的Region集合                            │
│  根据价值排序，优先回收垃圾最多的Region               │
└─────────────────────────────────────────────────────┘
```

## 三、G1 工作模式

### 整体流程

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Young GC  │ ──→ │  Young GC   │ ──→ │  Mixed GC   │
│   (只有新生代)│     │  (达到阈值)  │     │ (新生代+部分老年代)│
└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                        ┌──────────────────────┘
                        ↓
                ┌─────────────┐
                │  Full GC    │  ← 当 Mixed GC 无法跟上分配速度
                │  (兜底方案)   │     触发，单线程，长时间停顿
                └─────────────┘
```

### 1. Young GC（新生代回收）

```
触发条件：Eden 区满

┌─────────────────────────────────────────────────┐
│                  Young GC 流程                   │
├─────────────────────────────────────────────────┤
│ 1. 扫描 GC Roots                                  │
│ 2. 更新 RSet                                      │
│ 3. 复制存活对象到 Survivor 或 Old                 │
│ 4. 清空 Eden 和原 Survivor                        │
└─────────────────────────────────────────────────┘
```

### 2. 并发标记周期（Concurrent Marking）

```
当老年代占用达到 IHOP 阈值（默认45%）时触发

┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ 初始标记      │ → │ 并发标记      │ → │ 最终标记      │
│ (STW)        │   │ (并发)       │   │ (STW)        │
└──────────────┘   └──────────────┘   └───────┬──────┘
                                              ↓
                                      ┌──────────────┐
                                      │ 筛选回收      │
                                      │ (STW)        │
                                      └──────────────┘
```

| 阶段 | 说明 | STW |
|------|------|-----|
| 初始标记 | 标记 GC Roots 直接关联的对象 | 是 |
| 并发标记 | 从 GC Roots 遍历整个堆 | 否 |
| 最终标记 | 处理 SATB 队列中的变更 | 是 |
| 筛选回收 | 统计各 Region 垃圾价值 | 是 |

**SATB (Snapshot-At-The-Beginning)：**
- 并发标记开始时的对象快照
- 保证标记过程中新建对象不会被错误回收

### 3. Mixed GC

```
┌─────────────────────────────────────────────────┐
│                 Mixed GC                         │
├─────────────────────────────────────────────────┤
│ 回收范围：所有新生代 + 部分老年代                  │
│ 选择标准：垃圾最多的 Region（收益最大）            │
│ 停顿控制：根据 -XX:MaxGCPauseMillis 目标选择      │
└─────────────────────────────────────────────────┘
```

## 四、G1 核心特性

### 1. 可预测停顿时间模型

```bash
# 设置目标停顿时间（默认200ms）
-XX:MaxGCPauseMillis=200
```

**工作原理：**
- G1 维护每个 Region 的回收价值（垃圾量/回收时间）
- 根据目标停顿时间，选择收益最大的 Region 组成 CSet
- 保证 GC 停顿在预期范围内

### 2. 空间压缩（无碎片）

```
传统收集器：             G1：
┌─┬─┬─┬─┬─┬─┬─┐         ┌─┬─┬─┬─┬─┬─┬─┐
│█│ │█│ │ │█│ │   →    │█│█│█│ │ │ │ │
└─┴─┴─┴─┴─┴─┴─┘         └─┴─┴─┴─┴─┴─┴─┘
   碎片化                   整理后

█ = 存活对象
```

### 3. 大对象处理

```java
// 大对象判定标准
对象大小 > RegionSize * 50% → Humongous Region

// 示例：Region = 4MB
// 对象 > 2MB → Humongous
```

**处理策略：**
- 分配在连续的 Humongous Region
- 在并发标记阶段回收
- 避免大对象在新生代复制开销

## 五、关键参数配置

```bash
# 基础配置
-XX:+UseG1GC                    # 启用G1（JDK9+默认）
-Xms4g -Xmx4g                   # 堆大小（建议相等）
-XX:MaxGCPauseMillis=200        # 目标停顿时间

# Region 大小
-XX:G1HeapRegionSize=4m         # Region大小（1~32MB）

# 新生代比例（建议不手动设置，让G1自适应）
-XX:G1NewSizePercent=5          # 新生代最小占比（默认5%）
-XX:G1MaxNewSizePercent=60      # 新生代最大占比（默认60%）

# 并发标记触发阈值
-XX:InitiatingHeapOccupancyPercent=45  # 老年代占用阈值

# GC 日志
-Xlog:gc*:file=gc.log:time,level,tags
```

## 六、G1 vs 其他收集器

| 特性 | CMS | G1 | ZGC |
|------|-----|-----|-----|
| JDK版本 | 已废弃 | JDK9+默认 | JDK15+ |
| 目标停顿 | ~100ms | <200ms | <10ms |
| 内存碎片 | 有 | 无 | 无 |
| 支持堆大小 | <8GB | 4GB~32GB | >16GB |
| 并发整理 | 否 | 是 | 是 |
| 适用场景 | 老项目 | 通用服务端 | 大内存低延迟 |

## 七、G1 调优建议

### 1. 何时选择 G1？

```
✅ 推荐场景：
- 堆大小 6GB~32GB
- 需要可预测的停顿时间
- 存在内存碎片问题
- JDK 9+

❌ 不推荐场景：
- 堆太小（<4GB）
- 追求极致低延迟（选ZGC）
- 追求极致吞吐量（选Parallel）
```

### 2. 调优步骤

```
1. 设置合理堆大小
   -Xms -Xmx（相等，避免动态扩容）

2. 设置停顿目标
   -XX:MaxGCPauseMillis=200

3. 观察GC日志，调整IHOP
   -XX:InitiatingHeapOccupancyPercent=45

4. 若频繁Full GC：
   - 增大堆内存
   - 降低IHOP阈值
   - 检查是否有内存泄漏
```

### 3. 常见问题排查

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| 频繁 Full GC | 堆不足/IHOP不合理 | 增大堆/调整IHOP |
| 停顿超预期 | Region过大/小 | 调整G1HeapRegionSize |
| 吞吐量下降 | 停顿目标过小 | 增大MaxGCPauseMillis |

## 八、总结

```
G1 核心优势：
┌────────────────────────────────────────┐
│ 1. Region 分区 → 灵活的回收粒度         │
│ 2. 可预测停顿 → 业务友好                │
│ 3. 复制算法 → 无内存碎片                │
│ 4. 并发标记 → 低延迟                    │
└────────────────────────────────────────┘

适用场景：通用服务端应用，大内存场景的首选
```

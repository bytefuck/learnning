# JVM 技术面试题

## 一、内存结构（必问）

### 1. JVM 内存结构由哪些部分组成？
**答案：**
```
┌─────────────────────────────────────────┐
│              JVM 内存结构                │
├─────────────────────────────────────────┤
│  堆          │ 线程共享，存储对象实例      │
│  方法区      │ 线程共享，存储类信息、常量池  │
├─────────────────────────────────────────┤
│  虚拟机栈    │ 线程私有，存储栈帧、局部变量  │
│  本地方法栈  │ 线程私有，Native方法调用     │
│  程序计数器  │ 线程私有，记录执行位置       │
└─────────────────────────────────────────┘
```

### 2. 堆内存如何划分？
**答案：**
```
堆
├── 新生代 (Young) - 1/3
│   ├── Eden区 (8/10)
│   ├── Survivor0 (1/10)
│   └── Survivor1 (1/10)
└── 老年代 (Old) - 2/3
```
- 新对象在Eden创建
- Survivor用于复制算法中转
- 大对象或长期存活对象进入老年代

## 二、垃圾回收（核心）

### 3. 如何判断对象可以被回收？
**答案：**
| 方式 | 说明 |
|------|------|
| 引用计数法 | 有引用+1，引用失效-1；无法解决循环引用 |
| 可达性分析 | 从GC Roots出发，不可达则可回收（主流） |

**GC Roots包括：**
- 虚拟机栈中引用的对象
- 方法区静态属性引用
- 方法区常量引用
- 本地方法栈JNI引用

### 4. 垃圾回收算法有哪些？
**答案：**

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 标记-清除 | 标记后直接清除 | 简单 | 内存碎片 |
| 标记-整理 | 清除后整理内存 | 无碎片 | 效率较低 |
| 复制算法 | 分两块，存活复制到另一块 | 效率高 | 内存利用率低 |
| 分代收集 | 新生代复制，老年代标记整理 | 综合优势 | - |

### 5. 常见垃圾收集器对比？
**答案：**
```
新生代收集器：Serial、ParNew、Parallel Scavenge
老年代收集器：CMS、Serial Old、Parallel Old
整代收集器：G1、ZGC
```

| 收集器 | 特点 | 适用场景 |
|--------|------|----------|
| Serial | 单线程，STW | 客户端模式 |
| ParNew | Serial多线程版 | 配合CMS |
| Parallel Scavenge | 吞吐量优先 | 后台计算任务 |
| CMS | 并发收集，低延迟 | 已废弃 |
| G1 | 分Region，可预测停顿 | JDK9+默认 |
| ZGC | 并发整理，<10ms停顿 | JDK15+，大内存 |

## 三、类加载机制

### 6. 类加载过程是怎样的？
**答案：**
```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
         └───────┬───────┘
              连接阶段
```
- **加载**：读取.class文件到内存
- **验证**：确保字节码合法
- **准备**：为静态变量分配内存，赋零值
- **解析**：符号引用→直接引用
- **初始化**：执行<clinit>方法

### 7. 双亲委派模型是什么？为什么要这样设计？
**答案：**
```
        Bootstrap ClassLoader (核心类库)
                ↑
        Extension ClassLoader (扩展类库)
                ↑
        Application ClassLoader (应用类)
```
**工作流程：** 先委托父类加载，父类无法加载才自己加载

**设计目的：**
1. 安全性：防止核心API被篡改
2. 避免重复加载：父类已加载，子类不重复

**打破场景：** SPI机制（JDBC）、Tomcat类隔离

## 四、JVM调优

### 8. 常用JVM参数有哪些？
**答案：**
```bash
# 内存设置
-Xms512m          # 初始堆大小
-Xmx2g            # 最大堆大小
-Xmn256m          # 新生代大小
-XX:MetaspaceSize=128m   # 元空间大小

# GC相关
-XX:+UseG1GC      # 使用G1收集器
-XX:MaxGCPauseMillis=200 # 最大GC停顿
-XX:+PrintGCDetails      # 打印GC详情

# OOM处理
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/dump.hprof
```

### 9. 如何排查OOM问题？
**答案：**
1. 获取堆转储文件（jmap或启动参数）
2. 使用MAT/VisualVM分析
3. 查找大对象或泄漏点
4. 检查是否内存不足或资源未释放

## 五、字节码与编译

### 10. JVM如何执行字节码？
**答案：**
```
.java → 编译器 → .class → 类加载器 → JVM
                                    ↓
                            解释器执行
                               或
                         JIT编译成本地代码
```
- **解释执行**：逐条解释字节码
- **JIT编译**：热点代码编译成本地机器码（热点探测：方法调用计数器、回边计数器）

### 11. 有哪些JIT编译器？
**答案：**
| 编译器 | 特点 |
|--------|------|
| C1 (Client) | 快速编译，优化较少，启动快 |
| C2 (Server) | 深度优化，编译慢，运行快 |
| Graal | JDK10+，基于Graal VM |

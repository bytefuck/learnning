# 线程池实战场景问题

## 1. 如果线程池队列满了但还在不断提交任务，会发生什么？

### 任务提交流程回顾

```
提交任务
    |
    v
线程数 < corePoolSize?
    |
    +-- 是 --> 创建核心线程
    |
    +-- 否 --> 队列未满?
                |
                +-- 是 --> 加入队列
                |
                +-- 否 --> 线程数 < maximumPoolSize?
                            |
                            +-- 是 --> 创建非核心线程
                            |
                            +-- 否 --> 执行拒绝策略
```

### 队列满后的处理

**情况1：线程数 < maximumPoolSize**
- 创建非核心线程处理任务
- 非核心线程数量 = maximumPoolSize - corePoolSize

**情况2：线程数 >= maximumPoolSize**
- 执行拒绝策略

### 不同拒绝策略的行为

**1. AbortPolicy（默认）**
```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    5, 10, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(100),
    new ThreadPoolExecutor.AbortPolicy()
);

// 队列满且线程数=10时，抛出异常
executor.execute(task);
// 抛出: RejectedExecutionException
```

**2. CallerRunsPolicy**
```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    5, 10, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(100),
    new ThreadPoolExecutor.CallerRunsPolicy()
);

// 队列满时，由提交任务的线程执行
executor.execute(task);
// 在调用者线程中同步执行
```

**效果**：
- 可以减缓任务提交速度
- 相当于一种背压机制

**3. DiscardPolicy**
```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    5, 10, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(100),
    new ThreadPoolExecutor.DiscardPolicy()
);

// 队列满时，静默丢弃任务
executor.execute(task);
// 任务被丢弃，无任何提示
```

**4. DiscardOldestPolicy**
```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    5, 10, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(100),
    new ThreadPoolExecutor.DiscardOldestPolicy()
);

// 队列满时，丢弃最老的任务，重新提交
executor.execute(task);
// 队列头部任务被丢弃，当前任务加入队列
```

### 实战建议

**1. 使用有界队列**
```java
// 推荐：指定队列容量
new ArrayBlockingQueue<>(1000)
new LinkedBlockingQueue<>(1000)

// 不推荐：无界队列
new LinkedBlockingQueue<>()  // 容量为 Integer.MAX_VALUE
```

**2. 监控队列状态**
```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(...);

// 监控任务
executor.execute(() -> {
    // 检查队列大小
    int queueSize = executor.getQueue().size();
    int activeCount = executor.getActiveCount();
    long completedTasks = executor.getCompletedTaskCount();
    
    // 告警
    if (queueSize > 800) {
        alert("Queue almost full: " + queueSize);
    }
});
```

**3. 自定义拒绝策略**
```java
public class LogAndStorePolicy implements RejectedExecutionHandler {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        // 记录日志
        log.warn("Task rejected, queue size: {}", e.getQueue().size());
        
        // 持久化任务
        saveToDatabase(r);
        
        // 发送告警
        sendAlert("Thread pool rejected task");
    }
}
```

## 2. 如何监控线程池的运行状态？

### 线程池监控指标

| 指标 | 方法 | 说明 |
|------|------|------|
| 核心线程数 | getCorePoolSize() | 核心线程数 |
| 最大线程数 | getMaximumPoolSize() | 最大线程数 |
| 当前线程数 | getPoolSize() | 当前工作线程数 |
| 活跃线程数 | getActiveCount() | 正在执行任务的线程数 |
| 完成任务数 | getCompletedTaskCount() | 已完成任务数 |
| 任务总数 | getTaskCount() | 提交的任务总数 |
| 队列大小 | getQueue().size() | 队列中等待的任务数 |

### 监控实现方式

**1. 定时监控**
```java
public class ThreadPoolMonitor {
    private final ThreadPoolExecutor executor;
    private final ScheduledExecutorService monitorExecutor;
    
    public ThreadPoolMonitor(ThreadPoolExecutor executor) {
        this.executor = executor;
        this.monitorExecutor = Executors.newSingleThreadScheduledExecutor();
    }
    
    public void start() {
        monitorExecutor.scheduleAtFixedRate(() -> {
            log.info("Pool: {}, Active: {}, Queue: {}, Completed: {}, Task: {}",
                executor.getPoolSize(),
                executor.getActiveCount(),
                executor.getQueue().size(),
                executor.getCompletedTaskCount(),
                executor.getTaskCount());
        }, 0, 1, TimeUnit.SECONDS);
    }
}
```

**2. 继承 ThreadPoolExecutor**
```java
public class MonitoredThreadPoolExecutor extends ThreadPoolExecutor {
    
    public MonitoredThreadPoolExecutor(int corePoolSize, int maximumPoolSize,
            long keepAliveTime, TimeUnit unit, BlockingQueue<Runnable> workQueue) {
        super(corePoolSize, maximumPoolSize, keepAliveTime, unit, workQueue);
    }
    
    @Override
    protected void beforeExecute(Thread t, Runnable r) {
        super.beforeExecute(t, r);
        log.info("Task start: {}", r);
    }
    
    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        super.afterExecute(r, t);
        if (t != null) {
            log.error("Task exception: {}", t.getMessage());
        }
        log.info("Task end: {}", r);
    }
    
    @Override
    protected void terminated() {
        super.terminated();
        log.info("Thread pool terminated");
    }
}
```

**3. 使用动态代理**
```java
public class ThreadPoolProxy implements ExecutorService {
    private final ExecutorService target;
    
    public ThreadPoolProxy(ExecutorService target) {
        this.target = target;
    }
    
    @Override
    public void execute(Runnable command) {
        long start = System.currentTimeMillis();
        target.execute(() -> {
            try {
                command.run();
            } finally {
                long cost = System.currentTimeMillis() - start;
                log.info("Task cost: {}ms", cost);
            }
        });
    }
    
    // ... 其他方法代理
}
```

## 3. 线程池中出现 OOM 或任务执行异常如何排查？

### OOM 问题排查

**1. 线程数过多导致的 OOM**

**原因**：
- maximumPoolSize 设置过大
- 使用 CachedThreadPool（maximumPoolSize = Integer.MAX_VALUE）
- 任务执行时间过长，线程堆积

**排查方法**：
```bash
# 查看线程数量
jstack <pid> | grep "java.lang.Thread.State" | wc -l

# 查看线程详情
jstack <pid> > thread_dump.txt

# 查看堆内存使用
jmap -heap <pid>

# 导出堆内存
jmap -dump:live,format=b,file=heap.hprof <pid>
```

**解决方案**：
```java
// 限制最大线程数
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,    // 核心线程数
    50,    // 最大线程数（合理设置）
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000)
);
```

**2. 任务队列过大导致的 OOM**

**原因**：
- 使用无界队列（LinkedBlockingQueue 无参构造）
- 任务消费速度 < 生产速度
- 队列任务堆积

**排查方法**：
```java
// 监控队列大小
public void checkQueue(ThreadPoolExecutor executor) {
    BlockingQueue<Runnable> queue = executor.getQueue();
    System.out.println("Queue size: " + queue.size());
    System.out.println("Queue remaining: " + queue.remainingCapacity());
}
```

**解决方案**：
```java
// 使用有界队列
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10, 50, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000),  // 指定容量
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);
```

### 任务执行异常排查

**1. 异常被吞掉的问题**

**原因**：
- submit() 提交的任务，异常会被 Future.get() 抛出
- execute() 提交的任务，异常会被 UncaughtExceptionHandler 处理

**示例**：
```java
ExecutorService executor = Executors.newFixedThreadPool(1);

// 使用 submit，异常被封装
Future<?> future = executor.submit(() -> {
    throw new RuntimeException("Error!");
});
try {
    future.get();  // 这里抛出 ExecutionException
} catch (ExecutionException e) {
    System.out.println(e.getCause());  // 获取原始异常
}

// 使用 execute，异常默认打印到控制台
executor.execute(() -> {
    throw new RuntimeException("Error!");
});
```

**2. 自定义异常处理**

```java
// 设置线程工厂
ThreadFactory factory = r -> {
    Thread t = new Thread(r);
    t.setUncaughtExceptionHandler((thread, throwable) -> {
        log.error("Thread {} error: {}", thread.getName(), throwable.getMessage());
    });
    return t;
};

ThreadPoolExecutor executor = new ThreadPoolExecutor(
    5, 10, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100),
    factory
);

// 或者重写 afterExecute
public class MyThreadPoolExecutor extends ThreadPoolExecutor {
    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        super.afterExecute(r, t);
        if (t != null) {
            log.error("Task exception: ", t);
        }
        // 处理 Future 的异常
        if (r instanceof Future<?>) {
            try {
                ((Future<?>) r).get();
            } catch (ExecutionException e) {
                log.error("Future exception: ", e.getCause());
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }
}
```

public class LoggingThreadPoolExecutor extends ThreadPoolExecutor {
    
    @Override
    public void execute(Runnable command) {
        long start = System.currentTimeMillis();
        super.execute(() -> {
            try {
                command.run();
            } catch (Exception e) {
                log.error("Task execution failed: ", e);
                throw e;
            } finally {
                log.info("Task cost: {}ms", System.currentTimeMillis() - start);
            }
        });
    }
    
    @Override
    protected void beforeExecute(Thread t, Runnable r) {
        log.info("Thread {} start task: {}", t.getName(), r);
    }
    
    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        if (t != null) {
            log.error("Task exception: ", t);
        }
    }
}

### 常见问题总结

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| OOM: unable to create new native thread | 线程数过多 | 限制最大线程数，使用有界队列 |
| OOM: Java heap space | 任务队列过大 | 使用有界队列，设置拒绝策略 |
| 任务执行无响应 | 任务阻塞或死锁 | 设置超时，监控任务执行时间 |
| 异常被吞掉 | 未正确处理异常 | 使用 afterExecute 或 UncaughtExceptionHandler |
| 线程池无法关闭 | 任务未响应中断 | 合理设计任务，正确处理中断 |

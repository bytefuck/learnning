# 面试宝典生成器

## 概述

这是一个帮助用户生成结构化、系统性面试问答文档的skill。针对用户输入的技术点，由浅入深地生成面试问题和详细答案。

## 工作流程

### 1. 理解技术点

首先分析用户输入的技术点：
- 明确技术领域的范围
- 识别相关联的技术栈
- 确定技术点的核心概念

### 2. 创建目录结构

在用户当前工作目录下创建面试宝典目录：

```
面试宝典-{技术名称}/
├── README.md           # 汇总索引文件
├── 01-基础问题.md
├── 02-进阶问题.md
└── 03-实战问题.md
```

### 3. 内容生成规则

#### 基础问题章节 (01-基础问题.md)

包含5-8个问题，覆盖：
- 核心概念定义
- 基本工作原理
- 主要特性与用途
- 与相似技术的对比
- 基本使用方式

每个问题回答要求：
- 定义清晰准确
- 配合简单示例说明
- 篇幅200-400字

#### 进阶问题章节 (02-进阶问题.md)

包含5-8个问题，覆盖：
- 内部实现机制
- 性能优化原理
- 高级特性使用
- 架构设计考量
- 源码层面的理解

每个问题回答要求：
- 深入原理层面
- 配合架构图或流程图（使用Mermaid）
- 源码分析（如适用）
- 篇幅400-800字

#### 实战问题章节 (03-实战问题.md)

包含5-8个问题，覆盖：
- 实际项目中的应用场景
- 常见问题排查
- 性能调优实践
- 生产环境最佳实践
- 系统设计题目

每个问题回答要求：
- 结合真实场景
- 包含具体代码示例
- 问题诊断思路
- 篇幅600-1000字

### 4. 汇总索引文件 (README.md)

创建总览文件，包含：

```markdown
# {技术名称} 面试宝典

## 目录

- [基础问题](./01-基础问题.md)
- [进阶问题](./02-进阶问题.md)
- [实战问题](./03-实战问题.md)

## 概述

{技术简介，100-200字}

## 知识图谱

{使用Mermaid绘制技术点的关系图}

## 学习建议

{针对不同水平的读者给出学习路径建议}
```

## 问题格式模板

每个问题使用以下格式：

```markdown
## Q{n}: {问题标题}

**问题**：{完整问题}

**答案**：

{详细解答}

{如需代码示例}

\`\`\`{语言}
// 代码示例
\`\`\`

{如需图表}

\`\`\`mermaid
graph TD
    A[概念] --> B[实现]
\`\`\`

**要点总结**：
- {要点1}
- {要点2}
- {要点3}

---
```

## 质量标准

1. **准确性**：所有技术内容必须准确无误
2. **深度**：进阶和实战问题要有足够深度
3. **实用性**：答案要能帮助读者真正理解
4. **结构化**：层次分明，便于阅读
5. **完整性**：覆盖该技术点的主要方面

## 执行步骤

### 第一步：创建目录和 README
```bash
mkdir -p {技术名称}
```
创建 README.md 汇总索引文件（包含目录、概述、知识图谱、学习建议）

### 第二步：生成基础问题 (01-基础问题.md)
直接写入文件，包含 5-8 个基础问题，每个问题 200-400 字

### 第三步：生成进阶问题 (02-进阶问题.md)
直接写入文件，包含 5-8 个进阶问题，每个问题 400-800 字，使用 Mermaid 图表

### 第四步：生成实战问题 (03-实战问题.md)
**重要**：由于实战问题内容较长（每个问题 600-1000 字），采用以下策略：
- 如果内容超过 30000 字，分批写入（每次写入 2-3 个问题）
- 或使用子 agent 协助生成，但必须明确要求使用 Write 工具直接写入文件
- 写入后立即读取验证文件是否存在

### 第五步：验证和告知
检查所有文件是否创建成功，告知用户文件位置和结构

## 文件写入规范

**必须遵守的写入规则**：

1. **使用 Write 工具**：每个章节文件使用一次 Write 工具完整写入，不要分多次编辑
2. **内容完整性**：确保写入的内容包含该章节所有问题，不要截断
3. **写入后验证**：写入后使用 Read 或 Bash 工具验证文件是否存在且大小合理
4. **避免任务委托丢失**：如使用子 agent 生成内容，必须明确要求其使用 Write 工具写入指定路径，并在完成后验证文件

## 大文件处理策略

当单个文件内容超过 50000 字符时：

```
方案 A：完整写入
- 使用 Write 工具一次性写入完整内容
- 适用于 50000-80000 字符

方案 B：分批写入
- 先写入前半部分（问题 1-4）
- 使用 Edit 工具追加后半部分（问题 5-8）
- 适用于 80000+ 字符

方案 C：子 agent 协助
- 明确要求子 agent 使用 Write 工具写入文件
- 子 agent 完成后，主 agent 验证文件存在
- 如文件不存在，主 agent 重新生成并写入
```

## 注意事项

- 如果技术点范围过大（如"Java"），建议用户聚焦到具体模块
- 可以根据技术点的复杂度调整问题数量
- 对于快速发展的技术，注意内容时效性
- 适当参考官方文档和权威资料
- **关键**：生成实战问题时，如使用子 agent，必须在任务结束后立即验证文件是否存在

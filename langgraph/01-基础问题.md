# LangGraph 基础问题

## Q1: 什么是 LangGraph？它的核心设计理念是什么？

**问题**：请介绍 LangGraph 是什么，以及它的核心设计理念。

**答案**：

LangGraph 是 LangChain 团队开发的一个库，专门用于构建有状态、多参与者的 LLM 应用。它将 AI 工作流建模为图结构（Graph），通过节点（Node）和边（Edge）来组织复杂的应用逻辑。

**核心设计理念**：

1. **图结构思维**：将工作流视为图，节点代表计算步骤（如调用 LLM、执行工具、处理数据），边代表控制流（顺序、条件分支、循环）

2. **状态管理**：引入共享状态（State）概念，所有节点都可以读取和修改状态，实现跨步骤的信息传递

3. **可组合性**：支持子图嵌套，可以将复杂工作流拆分为可复用的子图

4. **持久化支持**：通过 Checkpoint 机制实现状态持久化，支持长时间运行的任务和人机交互

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict

# 定义状态结构
class State(TypedDict):
    messages: list

# 创建图
graph = StateGraph(State)
graph.add_node("agent", agent_function)
graph.set_entry_point("agent")
graph.add_edge("agent", END)
```

**要点总结**：
- LangGraph 是基于图的 AI 应用框架
- 核心是节点、边、状态三个概念
- 适合构建需要维护状态的复杂应用
- 支持循环、条件分支等控制流

## Q2: LangGraph 中的 State 是什么？如何定义和使用？

**问题**：请解释 LangGraph 中 State 的概念，以及如何定义和使用它。

**答案**：

State（状态）是 LangGraph 的核心概念，它是一个在整个图执行过程中被共享和传递的数据结构。所有节点都可以读取和修改状态，实现节点间的信息传递。

**定义 State**：

State 通常使用 Python 的 TypedDict 或 Pydantic 模型定义：

```python
from typing import TypedDict, List, Annotated
from operator import add

# 方式 1: 使用 TypedDict
class State(TypedDict):
    messages: List[str]
    count: int

# 方式 2: 使用 Annotated 定义归约器
class State(TypedDict):
    # 新值会追加到列表末尾
    messages: Annotated[List[str], add]
    # 新值会覆盖旧值
    count: int
```

**使用 State**：

```python
def node_function(state: State) -> State:
    # 读取状态
    current_messages = state["messages"]
    
    # 修改状态
    new_message = "Hello from node"
    return {
        "messages": [new_message],  # 根据归约器处理
        "count": state["count"] + 1
    }

# 节点返回的字典会合并到全局状态
```

**状态归约器（Reducer）**：

- 默认行为：新值覆盖旧值
- 使用 `Annotated` 指定归约器：如 `add` 用于列表追加，`operator.add` 用于数值累加

**要点总结**：
- State 是跨节点共享的数据结构
- 使用 TypedDict 或 Pydantic 定义
- 通过归约器控制状态合并行为
- 节点返回的字典会合并到全局状态

## Q3: 如何在 LangGraph 中创建和编译一个图？

**问题**：请描述在 LangGraph 中创建和编译图的基本步骤。

**答案**：

创建和编译 LangGraph 图需要以下步骤：

**1. 定义状态结构**：

```python
from typing import TypedDict, List

class State(TypedDict):
    messages: List[str]
```

**2. 创建 StateGraph 实例**：

```python
from langgraph.graph import StateGraph, END

graph = StateGraph(State)
```

**3. 添加节点**：

```python
def node_a(state: State) -> State:
    return {"messages": state["messages"] + ["Node A"]}

def node_b(state: State) -> State:
    return {"messages": state["messages"] + ["Node B"]}

graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)
```

**4. 设置边和入口点**：

```python
graph.set_entry_point("node_a")  # 设置入口节点
graph.add_edge("node_a", "node_b")  # 添加边
graph.add_edge("node_b", END)  # 连接到结束点
```

**5. 编译图**：

```python
app = graph.compile()
```

**6. 运行图**：

```python
result = app.invoke({"messages": []})
print(result)
```

**完整示例**：

```python
from typing import TypedDict, List
from langgraph.graph import StateGraph, END

class State(TypedDict):
    messages: List[str]

def node_a(state: State) -> State:
    return {"messages": state["messages"] + ["Node A"]}

def node_b(state: State) -> State:
    return {"messages": state["messages"] + ["Node B"]}

graph = StateGraph(State)
graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)
graph.set_entry_point("node_a")
graph.add_edge("node_a", "node_b")
graph.add_edge("node_b", END)

app = graph.compile()
result = app.invoke({"messages": []})
# 输出：{"messages": ["Node A", "Node B"]}
```

**要点总结**：
- 创建 StateGraph 需先定义状态结构
- 使用 add_node 添加节点
- 使用 add_edge 连接节点
- 必须设置入口点和结束点
- 编译后才能执行

## Q4: LangGraph 中的节点（Node）和边（Edge）有什么区别？

**问题**：请解释 LangGraph 中节点和边的区别及其作用。

**答案**：

**节点（Node）**：

节点是执行实际计算的基本单元，代表工作流中的一个步骤。

```python
def agent_node(state: State) -> State:
    # 执行 LLM 调用
    response = llm.invoke(state["messages"])
    return {"messages": [response]}

graph.add_node("agent", agent_node)
```

**特点**：
- 接收当前状态作为输入
- 返回状态更新（字典）
- 可以调用 LLM、执行工具、处理数据等
- 节点名称在图中唯一

**边（Edge）**：

边定义节点之间的控制流，决定执行顺序。

```python
# 固定边：总是从 A 到 B
graph.add_edge("node_a", "node_b")

# 条件边：根据条件决定下一节点
graph.add_conditional_edges(
    "router",
    route_function,  # 返回下一节点名称
    {
        "continue": "agent",
        "end": END
    }
)
```

**特点**：
- 不执行计算，只控制流程
- 分为固定边和条件边
- 条件边基于函数返回值决定走向
- 支持循环（边指向之前的节点）

**区别对比**：

| 特性 | 节点 | 边 |
|------|------|-----|
| 作用 | 执行计算 | 控制流程 |
| 输入 | 状态 | 节点名称 |
| 输出 | 状态更新 | 下一节点名称 |
| 数量 | 可多个 | 连接节点 |
| 条件 | 无条件执行 | 可条件分支 |

**要点总结**：
- 节点负责计算，边负责流程控制
- 节点接收和返回状态
- 边决定下一执行节点
- 条件边实现分支逻辑
- 两者配合实现完整工作流

## Q5: 什么是条件边（Conditional Edges）？如何使用？

**问题**：请解释 LangGraph 中条件边的概念和使用方法。

**答案**：

条件边（Conditional Edges）允许根据运行时状态动态决定下一执行节点，实现分支逻辑。

**基本条件边**：

```python
from langgraph.graph import StateGraph, END

def router(state: State) -> str:
    # 根据状态决定下一节点
    if "error" in state:
        return "error_handler"
    else:
        return "normal_flow"

graph = StateGraph(State)
graph.add_node("router", router_function)
graph.add_node("error_handler", handle_error)
graph.add_node("normal_flow", normal_process)

graph.set_entry_point("router")
graph.add_conditional_edges(
    "router",           # 源节点
    router,             # 路由函数
    {                   # 映射关系
        "error_handler": "error_handler",
        "normal_flow": "normal_flow"
    }
)
```

**路由函数返回值**：

路由函数可以返回：
1. **字符串**：单个下一节点
2. **字符串列表**：多个并行节点

```python
# 返回单个节点
def router_single(state) -> str:
    return "node_a" if state["condition"] else "node_b"

# 返回多个节点（并行执行）
def router_parallel(state) -> List[str]:
    return ["node_a", "node_b"]
```

**使用场景**：

```python
# 场景 1: 错误处理
def route_or_error(state):
    if state.get("error"):
        return "error"
    return "continue"

# 场景 2: 意图路由
def route_by_intent(state):
    intent = classify_intent(state["query"])
    return f"handle_{intent}"

# 场景 3: 条件循环
def should_continue(state):
    if len(state["messages"]) < 10:
        return "continue"
    return "end"
```

**完整示例**：

```python
def should_continue(state: State) -> str:
    messages = state.get("messages", [])
    if len(messages) < 5:
        return "continue"
    return "end"

graph.add_node("agent", agent_function)
graph.add_node("should_continue", lambda s: s)

graph.set_entry_point("agent")
graph.add_edge("agent", "should_continue")
graph.add_conditional_edges(
    "should_continue",
    should_continue,
    {
        "continue": "agent",
        "end": END
    }
)
```

**要点总结**：
- 条件边实现动态分支逻辑
- 路由函数决定下一节点
- 支持返回单个或多个节点
- 常用于循环、错误处理、意图路由
- 使用 add_conditional_edges 添加

## Q6: LangGraph 如何支持循环（Loop）？

**问题**：请解释 LangGraph 中如何实现循环结构。

**答案**：

LangGraph 通过边指向已访问过的节点来实现循环，这是图结构的自然特性。

**基本循环模式**：

```python
graph = StateGraph(State)

graph.add_node("agent", agent_function)
graph.add_node("check", check_function)

graph.set_entry_point("agent")
graph.add_edge("agent", "check")

# 条件边创建循环
graph.add_conditional_edges(
    "check",
    should_continue,  # 返回 "agent" 或 "end"
    {
        "agent": "agent",  # 回到 agent 形成循环
        "end": END
    }
)
```

**循环控制函数**：

```python
def should_continue(state: State) -> str:
    messages = state.get("messages", [])
    
    # 终止条件
    if len(messages) >= 10:
        return "end"
    
    # 检查是否完成
    last_message = messages[-1]
    if is_complete(last_message):
        return "end"
    
    return "continue"  # 继续循环
```

**ReAct Agent 循环**：

典型的 Agent 模式（思考 - 行动 - 观察循环）：

```python
def agent(state):
    # 思考：决定下一步
    action = decide_action(state)
    return {"action": action}

def tools_node(state):
    # 行动：执行工具
    result = execute_tool(state["action"])
    return {"observation": result}

def check_done(state):
    # 检查是否完成
    if is_final_answer(state):
        return "end"
    return "continue"

graph.add_node("agent", agent)
graph.add_node("tools", tools_node)

graph.set_entry_point("agent")
graph.add_edge("agent", "tools")
graph.add_conditional_edges(
    "tools",
    check_done,
    {
        "continue": "agent",  # 循环回 agent
        "end": END
    }
)
```

**防止无限循环**：

```python
class State(TypedDict):
    messages: List[str]
    iteration_count: int

def check_limit(state: State) -> str:
    if state["iteration_count"] >= 10:  # 最大迭代次数
        return "end"
    return "continue"
```

**要点总结**：
- 通过边指向前序节点实现循环
- 条件边控制循环终止
- 需要明确的退出条件
- 建议添加最大迭代次数限制
- 典型应用：ReAct Agent 模式

## Q7: LangGraph 与 LangChain 是什么关系？

**问题**：请解释 LangGraph 和 LangChain 的关系及区别。

**答案**：

**关系**：

LangGraph 是 LangChain 生态系统的一部分，由同一团队开发，两者互补使用。

```
LangChain 生态系统
├── LangChain (核心库)
│   └── 提供 Chains, Agents, Tools, Memory 等
├── LangGraph (图结构框架)
│   └── 基于 LangChain 构建复杂工作流
├── LangSmith (可观测性平台)
└── LangServe (部署服务)
```

**集成使用**：

```python
from langchain.chat_models import ChatOpenAI
from langchain.tools import Tool
from langgraph.graph import StateGraph, END

# 使用 LangChain 的组件
llm = ChatOpenAI(model="gpt-4")
tools = [Tool(...)]

# 在 LangGraph 节点中使用
def agent_node(state):
    # 使用 LangChain 的 LLM
    response = llm.invoke(state["messages"])
    
    # 使用 LangChain 的工具
    tool_result = tools[0].invoke(response)
    
    return {"messages": [response]}
```

**区别对比**：

| 特性 | LangChain | LangGraph |
|------|-----------|-----------|
| 核心抽象 | Chain（链式） | Graph（图式） |
| 控制流 | 线性为主 | 支持循环、分支 |
| 状态管理 | Memory | 显式 State |
| 适用场景 | 简单任务 | 复杂工作流 |
| 学习曲线 | 较低 | 中等 |

**选择建议**：

```python
# 使用 LangChain 的场景
- 简单的 LLM 调用
- 线性的处理流程
- 快速的 Prototype

# 使用 LangGraph 的场景
- 需要循环的 Agent
- 复杂的状态管理
- 多人协作流程
- 需要 Human-in-the-loop
```

**要点总结**：
- LangGraph 是 LangChain 生态的补充
- 可以混合使用两者组件
- LangChain 适合简单线性流程
- LangGraph 适合复杂图式工作流
- 两者不是替代关系，而是互补

## Q8: LangGraph 支持哪些类型的图？

**问题**：请介绍 LangGraph 中支持的图类型及其特点。

**答案**：

LangGraph 提供多种预定义的图类型，适应不同场景：

**1. StateGraph（最常用）**：

通用的状态图，适用于大多数场景。

```python
from typing import TypedDict, List

class State(TypedDict):
    messages: List[str]
    metadata: dict

graph = StateGraph(State)
```

**特点**：
- 灵活定义任意状态结构
- 支持自定义归约器
- 适用于复杂状态管理

**2. MessageGraph**：

专门用于消息处理的图，状态固定为消息列表。

```python
from langgraph.graph import MessageGraph, END

graph = MessageGraph()

graph.add_node("agent", agent_function)
graph.set_entry_point("agent")
graph.add_edge("agent", END)

app = graph.compile()

# 直接传入消息列表
result = app.invoke([HumanMessage("Hello")])
```

**特点**：
- 状态预定义为消息列表
- 简化消息处理场景
- 适合对话系统

**3. 自定义图**：

通过继承 StateGraph 创建特定领域的图。

```python
class RAGGraph(StateGraph):
    def __init__(self):
        class RAGState(TypedDict):
            query: str
            documents: List[Document]
            answer: str
        super().__init__(RAGState)
```

**图类型对比**：

| 类型 | 状态定义 | 适用场景 |
|------|----------|----------|
| StateGraph | 自定义 | 通用场景 |
| MessageGraph | 固定为消息列表 | 对话系统 |
| 自定义图 | 领域特定 | 专用场景 |

**选择建议**：

```python
# 需要灵活的状态管理 → StateGraph
class State(TypedDict):
    messages: List
    metadata: Dict
    step: str

# 纯消息处理 → MessageGraph
graph = MessageGraph()

# 特定领域 → 自定义继承
class AgentGraph(StateGraph):
    ...
```

**要点总结**：
- StateGraph 最灵活，推荐使用
- MessageGraph 简化消息处理
- 可继承创建领域特定图
- 根据场景选择合适的图类型
- 状态设计决定图的复杂度
